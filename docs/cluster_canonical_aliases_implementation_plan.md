# План реализации канонических названий и алиасов каротажа

Документ описывает поэтапную реализацию механизма канонизации названий каротажей с хранением в существующей БД и UI-управлением из раздела скважин.

## Цель

Сделать управляемый и прозрачный процесс сопоставления `исходное название кривой -> каноническое название`, чтобы кластерный модуль работал только с каноническими параметрами.

## Базовые решения

- Хранение выполняется **в существующей БД** (без временного JSON-конфига в файлах).
- Точка входа в UI: раздел скважин, кнопка `aliac`.
- По кнопке `aliac` открывается отдельная форма управления каноническими названиями и алиасами.
- В кластеризации используются только канонические имена, а исходные названия нужны для трассировки и диагностики.

---

## Этап 1. Аудит текущей БД и фиксация структуры хранения

### 1.1. Выбрать фактические таблицы/поля
- По фактической ORM-схеме хранения в БД уже есть готовая структура:
  - **`canonical_well_log`** (модель `CanonicalWellLog`) для канонических названий:
    - `id` (PK);
    - `canonical_name` (`String`, `NOT NULL`, `UNIQUE`, `INDEX`);
    - `description` (`String`, nullable).
  - **`alias_well_log`** (модель `AliasWellLog`) для связей alias -> canonical:
    - `id` (PK);
    - `alias_name` (`String`, `NOT NULL`, `UNIQUE`, `INDEX`);
    - `canonical_id` (`Integer`, `NOT NULL`, `FK -> canonical_well_log.id`, `INDEX`).
- Приоритет alias в текущей схеме **не выделен отдельным полем** (и пока является опциональным расширением).
- Ограничения целостности уже частично обеспечены схемой:
  - уникальность canonical-имени и alias-имени;
  - невозможность alias без ссылки на canonical (`canonical_id NOT NULL`);
  - каскадное удаление alias на ORM-уровне через `relationship(..., cascade='all, delete-orphan')` у `CanonicalWellLog.aliases`.
- Для источника «всех доступных исходных названий кривых» использовать таблицу **`well_log`** (модель `WellLog`), поле **`curve_name`**.

### 1.2. Правила целостности данных
- Один alias может принадлежать только одному canonical.
- Удаление canonical не должно оставлять «висячие» ссылки.
- Проверка на дубли с учетом нормализации строки (trim/регистр).

### 1.3. Источник всех доступных исходных названий
- Список «всех возможных названий» брать из фактических названий кривых, присутствующих в БД (из таблицы каротажей).
- Сразу определить способ агрегации уникальных названий и их частотности (для UX).

**Критерий готовности этапа:** утверждена схема хранения в БД и правила целостности.

---

## Этап 2. Backend-сервис канонизации

### 2.1. CRUD для canonical
- Создание canonical.
- Переименование canonical.
- Удаление canonical.
- Получение списка canonical с агрегированной статистикой (кол-во алиасов, покрытие).

### 2.2. CRUD для alias
- Добавление alias в выбранный canonical.
- Удаление alias из canonical.
- Запрет добавления alias, если он уже связан с другим canonical.

### 2.3. Выдача справочных списков для формы
- Список alias, входящих в выбранный canonical.
- Список всех возможных названий кривых из БД.
- Список «нераспределенных» названий (не привязанных ни к одному canonical).

### 2.4. Резолвер имени
- Сервис `resolve_canonical(curve_name)`:
  - нормализация входного имени;
  - поиск canonical по alias;
  - возврат canonical или `None`.

**Критерий готовности этапа:** есть стабильный API для формы и для кластерного модуля.

## Этап 3. Форма управления (открывается кнопкой `aliac`)

### 3.1. Точка входа
- В разделе скважин добавить кнопку `aliac`.
- По нажатию открывать форму управления алиасами в модальном режиме.

### 3.2. Состав формы
Форма содержит 3 основные области:
1. **Список канонических названий**:
   - добавление;
   - удаление;
   - (опционально) переименование.
2. **Список алиасов выбранного canonical**:
   - добавление alias из списка доступных названий;
   - удаление alias.
3. **Список всех возможных названий из БД**:
   - поиск/фильтрация;
   - отображение частоты встречаемости;
   - визуальная отметка статуса (распределен / не распределен).

### 3.3. Подсветка нераспределенных названий
- Названия, которые не привязаны ни к одному canonical, подсвечивать отдельным цветом.
- Добавить быстрый фильтр «только нераспределенные».

### 3.4. UX-валидации
- Нельзя добавить пустой canonical/alias.
- Нельзя добавить дублирующий alias.
- При удалении canonical — подтверждение действия.
- После любого изменения форма мгновенно обновляет все 3 списка.

**Критерий готовности этапа:** пользователь может полностью управлять canonical/alias из формы `aliac`.

---

## Этап 4. Интеграция с модулем кластеризации

### 4.1. Добавление одного параметра
- При добавлении параметра брать canonical через резолвер.
- Если canonical не найден — показать понятное сообщение.

### 4.2. ADD ALL
- Формировать список параметров по canonical.
- Считать покрытие по скважинам на уровне canonical-параметров.

### 4.3. COLLECT
- Сбор выполнять по canonical-параметрам.
- Для диагностики сохранять источник фактического имени кривой (alias/original curve name).

### 4.4. Авто-инвалидация
- Изменение справочника alias/canonical инвалидирует собранные наборы, где это влияет на состав данных.

**Критерий готовности этапа:** кластерный модуль устойчиво работает на канонических именах и корректно реагирует на изменения справочника.

---

## Этап 5. Контроль качества и тест-кейсы

1. Создание/удаление/переименование canonical.
2. Добавление/удаление alias и запрет дублей/конфликтов.
3. Корректный список «всех названий из БД».
4. Подсветка и фильтр нераспределенных названий.
5. Корректная работа кнопки `aliac` и открытия формы.
6. `resolve_canonical` для известных/неизвестных названий.
7. Поведение `ADD ONE` и `ADD ALL` при полной и неполной канонизации.
8. Авто-инвалидация собранного набора после изменения alias-справочника.

---

## Порядок реализации

1. Этап 1: подтвердить схему БД и ограничения.
2. Этап 2: реализовать backend-сервис.
3. Этап 3: реализовать форму `aliac` и UX-подсветку.
4. Этап 4: встроить в кластерный pipeline.
5. Этап 5: тесты и стабилизация.

Такой порядок позволяет сначала закрепить источник истины в БД, затем безопасно нарастить UI и интеграцию без повторных переделок.
