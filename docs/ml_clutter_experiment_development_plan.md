# План разработки экспериментального блока ML Clutter для очистки воздушных помех

## 1. Назначение документа

Документ описывает поэтапный план разработки экспериментального блока `ML Clutter` для исследования ML-очистки георадарных профилей от воздушных помех. Экспериментальный блок должен запускаться из главного окна приложения по кнопке `ML Clutter`, открывать отдельное окно управления экспериментом, позволять выбирать текущий профиль, генерировать синтетические помехи, использовать реальные зашумлённые радарограммы как паттерны шума, обучать модель очистки и визуализировать все ключевые преобразования в основном окне `MainWindow`.

Основная исследовательская идея:

1. Берём чистую радарограмму `X_clean` в амплитудном диапазоне `0..256`.
2. Генерируем или извлекаем шумовой паттерн `N_noise` в том же диапазоне; по умолчанию извлечение паттерна сохраняет исходную шкалу амплитуд, без standard-нормализации.
3. Пользователь выбирает режим наложения real-pattern шума: основной режим — доминирующее амплитудное перекрытие, дополнительный legacy-режим — additive.
4. Для доминирующего перекрытия переводим `X_clean` и `N_noise` в диапазон `-128..+128`, в каждой точке выбираем значение с большим модулем и возвращаем результат в `0..256`.
5. Обучаем модель по основной паре `input: X_noisy`, `target: X_clean`; clutter/mask остаются диагностическими артефактами, а не обязательными target для baseline-обучения.
6. Применяем модель к синтетическим и реальным профилям.
7. Оцениваем, насколько хорошо модель удаляет воздушную помеху и насколько бережно сохраняет полезные подповерхностные отражения.

Ключевое расширение относительно первоначальной схемы: генератор должен поддерживать не только аналитические синтетические помехи, но и реальные радарограммы с шумом, загружаемые как библиотека паттернов воздушных помех. Это нужно, чтобы уменьшить риск расхождения между синтетическими и реальными помехами.

## 2. Общая архитектура блока

Экспериментальный блок состоит из трёх уровней:

1. UI-уровень.
2. Экспериментальная ML-логика.
3. Хранилище артефактов эксперимента.

### 2.1. UI-уровень

Планируемые компоненты:

- кнопка `ML Clutter` в главном окне;
- отдельное окно `ML Clutter Experiment`;
- вкладки настройки данных, генератора, паттернов шума, датасета, модели, обучения и инференса;
- передача результатов визуализации в основное окно `MainWindow`.

Окно эксперимента является панелью управления. Основное окно приложения используется как панель визуализации:

- основная радарограмма — отображение `clean`, `noise pattern`, `synthetic clutter`, `noisy`, `predicted clutter`, `cleaned`, `mask`, `error map`;
- график сигнала — сравнение отдельных трасс;
- график метрик — loss, SNR, RMSE, энергия вычитания;
- информационная панель — параметры эксперимента, логи, предупреждения и итоговые метрики.

Планируемый пакет:

## 3. Основные режимы работы блока

Блок `ML Clutter` должен поддерживать четыре режима.

### 3.1. Режим A: аналитическая синтетика

Чистый профиль зашумляется процедурно сгенерированными помехами:

- гиперболами;
- наклонными событиями;
- ringing;
- вертикальными всплесками;
- широкими зонами повышенной амплитуды;
- комбинациями нескольких типов.

Результат режима:

```text
X_clean + N_synthetic -> X_noisy
```

### 3.2. Режим B: реальные шумовые паттерны

Пользователь загружает реальные зашумлённые радарограммы. Из них извлекаются паттерны воздушных помех, которые затем накладываются на чистые профили.

Результат режима:

```text
X_clean + N_pattern_real -> X_noisy
```

### 3.3. Режим C: смешанный генератор

Генератор комбинирует аналитические помехи и реальные паттерны:

```text
N_clutter = a * N_synthetic + b * N_pattern_real
```

Этот режим должен стать основным для обучения, потому что он уменьшает synthetic-to-real gap.

### 3.4. Режим D: инференс на реальном профиле

Уже обученная модель применяется к профилю без ground truth:

```text
X_real_noisy -> model -> N_pred -> X_clean_pred
```

Результат оценивается визуально и через косвенные метрики: энергия вычитания, локальный RMS, карта уверенности, корреляция до/после.

## 5. Этап 2. Загрузка и валидация профилей

### 5.1. Задачи этапа

1. Реализовать выбор текущего профиля из приложения.
2. Реализовать загрузку профиля как `clean`.
3. Реализовать загрузку профиля как `real noisy`.
4. Добавить валидацию формы массива.
5. Добавить расчёт базовой статистики.
6. Добавить отображение профиля в главном окне.

### 5.2. Валидация

Минимальные проверки:

```python
profile.ndim == 2
profile.shape[1] == 512
np.isfinite(profile).all()
profile.shape[0] >= min_num_traces
```

Если профиль имеет форму `[512, num_traces]`, интерфейс должен предложить транспонировать его.

### 5.3. Статистика профиля

Для каждого профиля рассчитывать:

- число трасс;
- число отсчётов в трассе;
- min;
- max;
- mean;
- std;
- median;
- percentile 1/99;
- наличие NaN;
- наличие inf.

### 5.4. Ожидаемый результат этапа

Пользователь может выбрать профиль, нажать `Load as Clean` или `Load as Real Noisy`, увидеть статистику и отрисовку профиля в главном окне. Если профиль не подходит, пользователь получает понятное сообщение с причиной: неправильная размерность, не 512 отсчётов, NaN, inf или слишком мало трасс.

## 6. Этап 3. Предобработка и нормализация

### 6.1. Задачи этапа

1. Реализовать единый pipeline нормализации.
2. Поддержать несколько режимов нормализации.
3. Сохранять параметры нормализации в конфиге эксперимента.
4. Обеспечить обратное преобразование результата в исходный масштаб.

### 6.2. Режимы нормализации

#### 6.2.1. Standard normalization

```python
X_norm = (X - mean) / (std + eps)
```

#### 6.2.2. Robust normalization

```python
X_norm = (X - median) / (mad + eps)
```

#### 6.2.3. Percentile clipping + standardization

```python
X_clip = np.clip(X, p01, p99)
X_norm = (X_clip - mean) / (std + eps)
```

### 6.3. Дополнительные каналы

На этом этапе можно подготовить функции, но не обязательно включать их в первую модель:

- envelope через Hilbert transform;
- горизонтальный градиент;
- вертикальный градиент;
- локальный RMS;
- локальная энергия.

### 6.4. Ожидаемый результат этапа

Профиль можно привести к нормализованному виду и вернуть обратно в исходный масштаб без существенных искажений. Все параметры нормализации сохраняются в конфигурации эксперимента. В интерфейсе пользователь может включать или отключать режимы предобработки и видеть статистику до/после нормализации.

## 7. Этап 4. Аналитический генератор синтетических помех

### 7.1. Задачи этапа

1. Реализовать генерацию гиперболических событий.
2. Реализовать генерацию наклонных событий.
3. Реализовать ringing.
4. Реализовать вертикальные всплески.
5. Реализовать широкие шумовые зоны.
6. Реализовать маску помехи.
7. Реализовать смешивание до заданного SNR.

### 7.2. Выход генератора

Генератор должен возвращать:

```python
noisy, clutter, mask, meta
```

Где:

- `noisy` — зашумлённый профиль;
- `clutter` — добавленная помеха;
- `mask` — область влияния помехи;
- `meta` — параметры всех сгенерированных объектов.

### 7.3. Гиперболы

Гипербола задаётся параметрами:

- `x0`;
- `z0`;
- `curvature`;
- `width_x`;
- `amplitude`;
- `frequency`;
- `phase`;
- `polarity`;
- `decay`.

Импульс можно задавать wavelet-функцией, например Ricker wavelet.

### 7.4. Наклонные события

Линия задаётся:

- `x_start`;
- `x_end`;
- `z_start`;
- `slope`;
- `amplitude`;
- `thickness`;
- `frequency`.

### 7.5. Ringing

Ringing задаётся:

- начальной трассой;
- конечной трассой;
- начальным sample;
- частотой;
- затуханием;
- фазой;
- амплитудой.

### 7.6. Вертикальные всплески

Всплески задаются:

- набором трасс;
- шириной по горизонтали;
- типом профиля всплеска;
- амплитудой;
- вероятностью появления.

### 7.7. Ожидаемый результат этапа

Пользователь может выбрать чистый профиль, включить один или несколько типов аналитических помех, нажать `Generate Synthetic Clutter` и увидеть в `MainWindow` чистый профиль, добавленную помеху, маску и итоговый `noisy`-профиль. Для каждого запуска сохраняется `meta`, позволяющий воспроизвести результат при том же `seed`.

## 8. Этап 5. Загрузка реальных радарограмм с шумом как паттернов

### 8.1. Назначение этапа

Этот этап нужен для уменьшения главного риска эксперимента: синтетические помехи могут быть слишком непохожи на реальные. Поэтому в блок добавляется возможность загружать реальные радарограммы с воздушными помехами и использовать их как источник шумовых паттернов.

### 8.2. Задачи этапа

1. Добавить загрузку одного или нескольких `real noisy` профилей.
2. Добавить карточку каждого шумового профиля в библиотеку паттернов.
3. Реализовать интерактивное или полуавтоматическое выделение областей шума.
4. Реализовать извлечение patch-паттернов шума.
5. Реализовать нормализацию и каталогизацию паттернов.
6. Реализовать предпросмотр паттернов.
7. Реализовать сохранение и загрузку библиотеки паттернов.

### 8.3. Способы выделения паттернов

#### 8.3.1. Ручное выделение области

Пользователь визуально выбирает область на зашумлённой радарограмме:

```text
x_start, x_end, z_start, z_end
```

Эта область сохраняется как кандидат на шумовой паттерн.

Преимущество:

- максимальный контроль эксперта.

Недостаток:

- требует ручной работы.

#### 8.3.2. Полуавтоматическое выделение по энергии

Автоматически ищутся области с высокой локальной энергией:

```python
local_energy = moving_rms(profile)
candidate_mask = local_energy > threshold
```

Пользователь подтверждает или отклоняет найденные области.

#### 8.3.3. Выделение по остаточной компоненте

Если есть грубый фильтр фона, можно получить остаток:

```python
residual = profile - smooth_background(profile)
```

Из остатка выделяются высокоэнергетические структурные события.

#### 8.3.4. Импорт готовой маски

Если пользователь заранее подготовил маску шума, блок должен позволить загрузить её:

```text
profile.npy + mask.npy
```

### 8.4. Структура паттерна

Каждый паттерн должен хранить:

```python
{
    "pattern_id": "...",
    "source_profile": "...",
    "array": np.ndarray,
    "mask": np.ndarray,
    "bbox": [x_start, x_end, z_start, z_end],
    "normalization": {...},
    "stats": {...},
    "tags": ["pole", "powerline", "ringing"],
    "created_at": "...",
    "comment": "..."
}
```

### 8.5. Теги паттернов

Пользователь должен иметь возможность помечать паттерны:

- `pole`;
- `powerline`;
- `metal_fence`;
- `building`;
- `ringing`;
- `vertical_spike`;
- `unknown`.

Теги пригодятся для управляемой генерации датасета.

### 8.6. Ожидаемый результат этапа

Пользователь может загрузить реальную зашумлённую радарограмму, выделить на ней область воздушной помехи, сохранить эту область как шумовой паттерн, увидеть паттерн отдельно в `MainWindow` и добавить его в библиотеку. Библиотека паттернов сохраняется на диск и может быть повторно загружена в другом эксперименте.

## 9. Этап 6. Генератор на основе реальных паттернов

### 9.1. Задачи этапа

1. Реализовать выбор паттерна из библиотеки.
2. Реализовать случайный crop/resize паттерна.
3. Реализовать масштабирование амплитуды паттерна.
4. Реализовать случайный сдвиг по трассам и sample.
5. Реализовать деформации паттерна.
6. Реализовать наложение паттерна на чистый профиль.
7. Реализовать смешанный режим `Synthetic + Pattern`.

### 9.2. Трансформации паттернов

Чтобы один реальный паттерн не копировался слишком буквально, нужны аугментации:

- amplitude scaling;
- polarity flip;
- horizontal flip;
- time shift;
- trace shift;
- mild stretching по горизонтали;
- mild stretching по вертикали;
- random crop;
- random fade-in/fade-out;
- добавление слабого jitter;
- смешивание нескольких паттернов.

### 9.3. Наложение паттерна

Базовая формула для реальных noise-паттернов больше не является аддитивной. Clean и noise рассматриваются как амплитудные изображения в одной шкале `0..256`, поэтому real-pattern extraction по умолчанию должен сохранять raw amplitude. В UI должен быть явный выбор режима наложения: `dominant_amplitude` или `additive`.

Для режима `dominant_amplitude` размещённый паттерн накладывается на той же глубине, на которой он был извлечён: `z_start` берётся из `pattern.bbox` и только ограничивается пределами целевого clean-профиля. Формула:

```python
N_placed = place(pattern, x0, pattern.bbox.z_start)
M_noisy = place(pattern_mask, x0, z0)

X_clean_centered = X_clean - 128
N_centered = N_placed - 128
noise_wins = M_noisy > 0 and abs(N_centered) > abs(X_clean_centered)
X_noisy = where(noise_wins, N_placed, X_clean)
```

Такой режим означает, что шум не складывается с чистым сигналом, а перекрывает его только там, где модуль амплитуды шума больше модуля чистого сигнала. Legacy-режим `additive` можно оставить для сравнения, но он должен быть явно выбран пользователем. Если нужен residual для диагностики, он вычисляется постфактум:

```python
N_effective = X_noisy - X_clean
```

### 9.4. Масштабирование под target SNR

Для real-pattern режима `target_snr_db` масштабирует подготовленный noise-паттерн до этапа амплитудного перекрытия. После перекрытия фактический residual может отличаться от исходного паттерна, поэтому отчётный SNR нужно считать по эффективной разнице:

```python
beta = compute_beta_for_target_snr(X_clean_patch, pattern, target_snr_db)
X_noisy = dominant_amplitude_overlay(X_clean, beta * pattern, mask)
N_effective = X_noisy - X_clean
actual_snr_db = compute_snr_db(X_clean, N_effective)
```

### 9.5. Смешанный режим

В смешанном режиме компоненты сначала собираются в общий noise-профиль, а итоговый `X_noisy` формируется тем же амплитудным перекрытием:

```python
N_total = beta_1 * N_synthetic + beta_2 * N_pattern
X_noisy = dominant_amplitude_overlay(X_clean, N_total, M_total)
N_effective = X_noisy - X_clean
```

Должны сохраняться обе компоненты:

```python
meta["synthetic_clutter"]
meta["pattern_clutter"]
meta["total_clutter"]
```

### 9.6. Ожидаемый результат этапа

Пользователь может выбрать clean-профиль, выбрать один или несколько реальных шумовых паттернов, настроить силу и явный режим наложения (`dominant_amplitude`/`additive`), сгенерировать зашумлённый профиль и увидеть в `MainWindow`: clean, выбранный паттерн в исходной амплитудной шкале, размещённый паттерн на исходной глубине, effective clutter, noisy и dominance/effective mask. Генератор умеет работать как только с реальными паттернами, так и в смешанном режиме с аналитической синтетикой. Основная обучающая пара — `noisy -> clean`.

### 12.1. Задачи этапа

### 12.2. Loss-функции

Базовый loss:

```python
L_clean = L1(clean_pred, clean_true)
```

Residual loss остаётся опциональным только для отдельного residual-эксперимента. В базовом режиме `clutter_true` не обязателен, потому что обучение идёт по паре `noisy -> clean`.

Gradient loss:

```python
L_grad = L1(grad(clean_pred), grad(clean_true))
```

Регуляризация избыточного вычитания применяется только в residual-режиме. Для базового direct-clean режима итоговый loss может быть:

```text
L = L_clean + λ_grad * L_grad
```

## 14. Этап 11. Инференс на полном профиле

### 14.1. Задачи этапа

1. Реализовать sliding-window inference.
2. Реализовать усреднение перекрывающихся patch’ей.
3. Реализовать alpha-слайдер силы фильтра.
4. Реализовать mask-aware очистку.
5. Реализовать сохранение результата.

### 14.2. Sliding-window схема

```python
for x_start in range(0, num_traces - patch_width + 1, stride):
    patch = X[x_start:x_start + patch_width]
    pred = model(patch)
    output_sum[x_start:x_end] += pred * window
    weight_sum[x_start:x_end] += window

clutter_pred = output_sum / weight_sum
```

### 14.3. Финальная очистка

Без маски:

```python
cleaned = noisy - alpha * clutter_pred
```

С маской:

```python
cleaned = noisy - alpha * mask_pred * clutter_pred
```

Пользователь может выбрать модель и профиль, нажать `Run Inference`, увидеть predicted clutter и cleaned profile в `MainWindow`, изменить `alpha` и сразу увидеть более мягкую или более сильную очистку. Результат можно сохранить как новый профиль или как отдельный артефакт эксперимента.
