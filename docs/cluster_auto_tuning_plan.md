# План реализации автоподбора параметров кластерного анализа

## 1. Цель и ожидаемый результат

Реализовать в модуле `Cluster` режим автоматического подбора параметров (AUTO), который:

1. Подбирает лучшие настройки по трём внутренним метрикам качества:
   - Silhouette (больше — лучше)
   - Davies-Bouldin (меньше — лучше)
   - Calinski-Harabasz (больше — лучше)
2. Поддерживает два режима поиска:
   - **COARSE**: грубый быстрый подбор
   - **FINE**: грубый + точный локальный подбор вокруг лучших кандидатов
3. Показывает результаты подбора в `tableWidget_cluster_auto_result`.
4. Позволяет применить лучший найденный конфиг к текущим контролам и запустить обычный `CALC`.

---

## 2. Текущее состояние (база, которую переиспользуем)

В коде уже есть рабочие блоки пайплайна, которые нужно использовать без дублирования логики:

- Очистка и подготовка признаков (`clean_features`, `preprocess_features`)
- PCA (`apply_pca`)
- Кластеризация (`cluster_data`: kmeans/hdbscan/gmm)
- Оценка качества (`evaluate_clustering`: silhouette, DB, CH)
- Ручной расчёт (`calculate_cluster`)

Также в интерфейсе уже есть:

- Кнопка `AUTO`
- Радио-кнопки режима `COARSE` / `FINE`
- `tableWidget_cluster_auto_result` для вывода результатов

Задача: связать UI и backend автопоиска.

---

## 3. Архитектура решения

### 3.1. Принципы

1. Не писать один монолитный метод — разбить на небольшие функции.
2. `AUTO` работает как orchestration-функция, которая вызывает переиспользуемые шаги.
3. Логика сравнения кандидатов едина для всех алгоритмов.
4. Результаты каждого запуска кандидата сохраняются в структуру leaderboard.

### 3.2. Предлагаемые функции

1. `build_auto_search_space(mode: str) -> dict`
   - Формирует сетку параметров для COARSE/FINE.

2. `run_cluster_candidate(base_data, candidate) -> dict`
   - Выполняет pipeline для одного кандидата:
     - preprocess
     - pca
     - cluster
     - evaluate
   - Возвращает метрики, служебные показатели и статус.

3. `score_candidate(metrics, stats, weights) -> float | None`
   - Агрегирует метрики в единый score.

4. `rank_candidates(results) -> list[dict]`
   - Сортирует кандидатов по score и применяет tie-break.

5. `build_fine_search_space(top_results) -> list[dict]`
   - Строит локальные окрестности для FINE-этапа.

6. `run_auto_cluster_tuning(auto_mode: str) -> dict`
   - Главный orchestrator:
     - COARSE
     - (опционально) FINE
     - возврат `best_result`, `top_results`, `raw_results`.

7. `render_auto_results_table(results)`
   - Заполняет `tableWidget_cluster_auto_result`.

8. `apply_auto_result_to_ui(best_result)`
   - Перекладывает лучший конфиг в UI-контролы.

## 4. Поэтапная реализация

## Этап 0. Подготовка контракта данных

**Задачи:**

1. Ввести единый словарь-кандидат `candidate_config`:
   - `scaler_mode`
   - `pca_enabled`, `pca_mode`, `pca_value`
   - `method`
   - `method_params` (kmeans/hdbscan/gmm)
2. Ввести единый словарь результата `candidate_result`:
   - `candidate_id`
   - `candidate_config`
   - `metrics` (`silhouette`, `davies_bouldin`, `calinski_harabasz`)
   - `stats` (`n_clusters`, `noise_fraction`, `n_samples_eval`)
   - `score`
   - `status` (`ok` / `invalid` / `error`)
   - `error_text`

**Критерий готовности:**
- Есть типизированный (или строго документированный) формат входа/выхода для кандидата.

---

## Этап 1. Реализовать COARSE search space

**Задачи:**

1. Определить компактную coarse-сетку (пример):
   - scaler: `none`, `standard`, `robust`
   - pca:
     - off
     - variance_ratio: `0.85`, `0.90`, `0.95`
   - method:
     - kmeans: `k=2..8`
     - hdbscan: `min_cluster_size={10,20,40}`, `min_samples={3,5,10}`
     - gmm: `n_components=2..8`, `covariance_type={full,diag}`
2. Добавить защиту от слишком большого числа комбинаций (лимит, например 200).

**Критерий готовности:**
- Функция генерирует валидный список кандидатов и умеет писать размер сетки в лог/`set_info`.

---

## Этап 2. Реализовать прогон одного кандидата

**Задачи:**

1. Вынести из `calculate_cluster` переиспользуемые шаги (без UI-зависимостей).
2. Для каждого кандидата:
   - применить preprocess
   - опционально PCA
   - выполнить кластеризацию
   - оценить метрики
3. Добавить обработку невалидных случаев:
   - < 2 кластера после удаления шума
   - пустая выборка
   - ошибки алгоритма

**Критерий готовности:**
- Один кандидат стабильно возвращает `candidate_result` со статусом.

---

## Этап 3. Реализовать scoring и ранжирование

**Задачи:**

1. Ввести нормализацию метрик:
   - silhouette: напрямую в [0..1]-подобный диапазон
   - DB: инверсия (`db_inv = 1 / (1 + db)` или ранговая)
   - CH: rank-based нормализация среди кандидатов
2. Формула суммарного score:

   `score = w_sil * sil_norm + w_db * db_norm + w_ch * ch_norm`

   Рекомендуемые веса по умолчанию:
   - `w_sil = 0.4`
   - `w_db = 0.3`
   - `w_ch = 0.3`

3. Ограничения-штрафы:
   - высокий `noise_fraction`
   - слишком маленькие кластеры
   - слишком много/мало кластеров

4. Tie-break при равном score:
   - выше silhouette
   - ниже DB
   - ниже noise_fraction

**Критерий готовности:**
- Кандидаты получают воспроизводимый и объяснимый порядок.

---

## Этап 4. Реализовать FINE search

**Задачи:**

1. Взять top-K (например, 5) кандидатов из coarse.
2. Для каждого построить локальную окрестность:
   - kmeans: `k±1..2`, `n_init` вверх
   - hdbscan: `min_cluster_size` ± шаг, `min_samples` ± шаг
   - gmm: `n_components` ± шаг, альтернативный `covariance_type`
   - PCA: локально `±0.02` для `variance_ratio` или `±2` компоненты
3. Прогнать fine-кандидаты и объединить leaderboard.

**Критерий готовности:**
- В режиме FINE итоговый лучший кандидат не хуже coarse-best по score.

---

## Этап 5. Интеграция с интерфейсом

**Задачи:**

1. Подключить `ui.pushButton_clust_auto.clicked` к новому обработчику.
2. В обработчике читать режим:
   - если `radioButton_cluster_coarse_auto` → COARSE
   - если `radioButton_cluster_fine_auto` → FINE
3. Очищать и заполнять `tableWidget_cluster_auto_result`.
4. Писать этапный прогресс в `set_info`:
   - `Coarse: i/N`
   - `Fine: j/M`
5. После завершения:
   - показать best summary
   - применить best-конфиг к UI (или предложить применить)

**Критерий готовности:**
- Кнопка AUTO реально запускает подбор и заполняет таблицу результатов.

## Этап 7. Устойчивость и производительность

**Задачи:**

1. Добавить soft-timeout или лимит числа кандидатов.
2. Кэшировать промежуточные преобразования (`scaler+pca`) при одинаковых параметрах.
3. Защитить UI от “заморозки”:
   - минимум `QApplication.processEvents()` в цикле
   - в перспективе вынести в `QThread`/worker.
4. Логировать исключения по кандидату без остановки всего подбора.

**Критерий готовности:**
- AUTO устойчиво отрабатывает даже при частично невалидной сетке.

---

## Этап 8. Валидация качества

**Проверки:**

1. COARSE:
   - запускается и возвращает непустой leaderboard.
2. FINE:
   - запускает coarse + fine.
3. Стабильность:
   - повторный запуск с теми же параметрами даёт близкий rank (при фиксированных random_state).
4. Граничные случаи:
   - мало данных,
   - все точки в одном кластере,
   - HDBSCAN почти весь шум.

---

## Этап 9. Rollout-план

1. Сначала внедрить backend + заполнение таблицы (без авто-применения).
2. Затем включить auto-apply лучшего конфига.
3. Затем добавить тонкие настройки auto-search (веса, top-K, лимиты) при необходимости.

---

## 5. Риски и решения

1. **Риск:** метрики конфликтуют между собой.  
   **Решение:** ранговая нормализация + настраиваемые веса.

2. **Риск:** долгий перебор.  
   **Решение:** staged-search, лимит кандидатов, coarse-first.

3. **Риск:** неустойчивость из-за случайности.  
   **Решение:** фиксировать `random_state`, при необходимости multi-seed для top-кандидатов.

4. **Риск:** пользователь не понимает, почему выбран победитель.  
   **Решение:** таблица с метриками + пояснение score в `set_info`.

---

## 6. Критерии приёмки

Функциональные:

1. AUTO-кнопка запускает подбор в выбранном режиме COARSE/FINE.
2. Таблица результатов заполняется (top-N строк).
3. Лучший конфиг определяется автоматически и может быть применён к UI.
4. После применения можно выполнить обычный `CALC` без ручного повторного ввода параметров.

Нефункциональные:

1. Подбор устойчив к ошибкам отдельных кандидатов.
2. Интерфейс остаётся отзывчивым.
3. Итоги подбора интерпретируемы пользователю.

---

## 7. Что делать сразу после реализации

1. Добавить краткую инструкцию в пользовательскую документацию (как пользоваться COARSE/FINE).
2. Зафиксировать значения сетки и весов score по умолчанию.
3. Собрать 2–3 эталонных кейса для регрессионной проверки качества подбора.
