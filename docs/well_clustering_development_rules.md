# Правила разработки блока подготовки данных для кластерного анализа каротажа

Документ фиксирует согласованные в обсуждении правила для реализации интерфейса и логики подготовки данных скважинного каротажа.

## 1) Общая концепция интерфейса

- В верхней части модуля должен быть **переключаемый блок источника данных**:
  - `Георадарные данные`
  - `Скважинный каротаж`
- Переключение должно менять только блок подготовки данных, а не весь pipeline анализа.

## 2) Блок "Скважинный каротаж" (верхняя часть)

### 2.1 Список скважин в наборе

Обязательные элементы:
- список добавленных скважин;
- кнопка `Добавить выбранную скважину`;
- кнопка `Добавить все в радиусе R от выбранного объекта` (использовать уже существующий механизм выбора радиуса из блока скважин);
- кнопка `Удалить выбранные скважины`;
- кнопка очистки набора (рекомендуется).

Обязательная индикация по каждой скважине:
- статус интервала (`задан/не задан`);
- покрытие выбранными параметрами (например `8/10`);
- скважины без интервала должны подсвечиваться как проблемные.

### 2.2 Интервалы

Принятое решение:
- **дорабатывать существующую форму `form_well_log`**, а не делать отдельный интервал-редактор с нуля.

Поддерживаемые режимы задания интервала:
- вручную (top/bottom или depth + interval);
- по маркерам (`boundary`);
- от маркера + мощность (fixed window).

Требования:
- предусмотреть массовое применение интервала к выбранному набору скважин;
- обязательно валидировать интервал (top < bottom, в пределах кривой, ненулевой диапазон);
- отображать статус валидности интервала в списке скважин.

### 2.3 Список параметров (каротажей)

Проблема неполных данных должна обрабатываться явно и без "падения" набора.

Правила UI:
- показывать список параметров с покрытием по набору скважин (`N из M`);
- поддерживать пороги покрытия для включения параметра;
- если параметр отсутствует в одной/нескольких скважинах, это не должно блокировать работу всего набора;
- нужно показывать в списке скважин, какие именно параметры отсутствуют.

Рекомендуемое представление:
- матрица доступности (`скважины x параметры`) с цветовой индикацией.

## 3) Алиасы / разные названия одних и тех же каротажей

Принятое решение:
- хранение и настройка алиасов делать в модуле скважин;
- в блоке кластеризации использовать уже канонизированные имена.

Рекомендации по реализации:
- добавить в `form_well_log` управление "наборами названий каротажа для одного метода";
- структура набора: `канонический параметр -> список допустимых имен`;
- при сборке датасета выбирать доступный алиас по приоритету.

## 4) Конструктор признаков

Принятое решение:
- конструктор признаков выносится в **отдельную форму**.

### 4.1 Минимальный обязательный набор инструментов (MVP)

- арифметика: `+`, `-`, `*`, `/`;
- унарные преобразования: `log`, `abs`, `sqrt`;
- производная по глубине (`dX/dz`);
- сглаживание (rolling mean/median);
- нормировка:
  - z-score,
  - robust (median/IQR),
  - min-max (опционально, но желательно).

### 4.2 Дополнительные инструменты (после MVP)

- отношения и контрасты (`A/B`, `A-B`, нормированные разности);
- rolling std/variance;
- маски и пороговые бинарные признаки;
- предпросмотр признака на выбранной скважине до сохранения.

### 4.3 Требования к UX конструктора признаков

- список исходных кривых слева;
- область сборки формулы/операций в центре;
- список созданных признаков справа;
- для каждого созданного признака показывать покрытие по скважинам и ошибки расчета.

## 5) Правила устойчивости данных (чтобы набор не "вылетал")

Должны быть явные параметры сборки:
- порог покрытия параметра;
- порог покрытия скважины;
- стратегия обработки пропусков:
  - исключение точек,
  - импутация по скважине,
  - импутация по набору;
- предзапусковая проверка объема выборки (если данных мало — понятное предупреждение и блок запуска).

## 6) Границы ответственности модулей

- `form_well_log`: загрузка/просмотр кривых, интервалы, маркеры, алиасы названий;
- `feature constructor` (новая форма): расчётные признаки и их валидация;
- верхний блок кластера: формирование набора скважин, контроль покрытия и запуск подготовки датасета.

## 7) Что уже есть в проекте и должно быть переиспользовано

- существующие элементы `form_well_log` для задания глубины/интервала и визуального контроля;
- существующий механизм работы с маркерами (`boundary`);
- существующая логика выбора вариантов названий каротажей (окно `MAP`) как база для алиасов;
- существующий выбор скважин по пространственному контексту объекта.

## 8) Принципы дальнейшей реализации

- сначала MVP без миграций БД, используя текущую модель;
- ввод новых сущностей БД только после стабилизации UX;
- все спорные правила (например, дефолтные пороги покрытия, порядок приоритета алиасов) фиксировать отдельным коротким RFC перед внедрением.
