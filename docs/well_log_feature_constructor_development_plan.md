# Поэтапный план разработки конструктора/калькулятора признаков Well Log для кластерного анализа

Документ фиксирует детальный план разработки конструктора расчётных признаков для вкладки **Well Log** в модуле кластерного анализа. Каждый этап планируется выполнять в отдельной git-ветке и завершать отдельным pull request.

## 0. Цель и принятые архитектурные решения

### 0.1. Цель разработки

Добавить в Well Log вкладку кластерного анализа отдельный конструктор признаков, запускаемый по кнопке **CONSTR**, который позволит:

1. создавать глобальные расчётные признаки на базе канонических каротажных кривых;
2. сохранять определения расчётных признаков в существующей таблице `feature_calculator`;
3. переиспользовать созданные признаки в разных Well Log datasets;
4. добавлять выбранные расчётные признаки в текущий Well Log dataset через существующую связь `cluster_well_log_parameter_from_calculator`;
5. рассчитывать эти признаки при `COLLECT` вместе с обычными canonical-параметрами;
6. строго блокировать сборку набора, если хотя бы один выбранный расчётный признак не может быть рассчитан для текущего dataset.

### 0.2. UI-точка входа

Конструктор открывается из Well Log вкладки кластерного анализа по кнопке:

```text
CONSTR
```

Кнопка должна располагаться рядом с блоком управления каротажными параметрами dataset, то есть рядом с кнопками добавления/удаления Well Log параметров. Назначение кнопки:

- открыть отдельную форму конструктора;
- передать в форму контекст текущего Well Log dataset;
- дать пользователю доступ к глобальной библиотеке расчётных признаков;
- позволить добавить сохранённый расчётный признак в текущий dataset.

### 0.3. Принятые решения

| Вопрос | Решение для MVP |
|---|---|
| Интерполяция кривых | Не выполняется |
| Несовпадение шага или глубинной сетки входных кривых | Ошибка, расчётный признак нельзя использовать |
| Нормировка в калькуляторе | Только внутри одной скважины |
| Нормировка по умолчанию | По всей скважине |
| Дополнительный вариант нормировки | Только по выбранному интервалу |
| Нормировка по всему dataset | Не входит в калькулятор, остаётся в предобработке кластерного анализа |
| Расчётные признаки | Глобальные, переиспользуемые между datasets |
| Если признак не считается | Блокировать добавление/`COLLECT` |
| Редактирование существующего признака | Не делать в MVP |
| Обработка зашкал/выбросов | В MVP без коррекции, но заложить параметр `outlier_policy` |
| Запуск конструктора | По кнопке `CONSTR` |

### 0.4. Минимальный набор операций MVP

Конструктор должен поддерживать:

1. арифметику:
   - `+`;
   - `-`;
   - `*`;
   - `/`;
2. унарные преобразования:
   - `log(X)`;
   - `abs(X)`;
   - `sqrt(X)`;
3. производную по глубине:
   - `dX/dz`;
4. сглаживание:
   - `rolling_mean(X, window)`;
   - `rolling_median(X, window)`;
5. нормировки внутри скважины:
   - `z-score`;
   - `robust` через median/IQR;
   - `min-max`.

### 0.5. Границы ответственности

#### `form_well_log`

Остаётся ответственным за:

- загрузку и просмотр кривых;
- работу с интервалами;
- маркеры и визуальный контроль;
- управление алиасами/каноническими именами, если этот функционал вызывается из модуля скважин.

#### Форма конструктора признаков

Отвечает за:

- создание новых глобальных расчётных признаков;
- валидацию формул и параметров операций;
- отображение глобальной библиотеки признаков;
- отображение применимости признака к текущему dataset;
- добавление признака в текущий dataset;
- удаление признака из текущего dataset;
- просмотр определения признака.

#### Well Log вкладка кластерного анализа

Отвечает за:

- формирование набора скважин;
- выбор интервалов;
- выбор обычных canonical-параметров;
- запуск формы конструктора по кнопке `CONSTR`;
- запуск `COLLECT`;
- индикацию актуальности собранного dataset;
- запуск дальнейшей кластеризации.

#### Backend-сервис калькулятора

Отвечает за:

- безопасный расчёт операций;
- строгую проверку глубинных сеток;
- расчёт coverage;
- подготовку ошибок в человекочитаемом виде;
- интеграцию расчётных признаков в `COLLECT`.

# Этап 1. Документирование RFC, добавление UI-точки входа и технического каркаса

## Ветка

```text
feature/well-log-constructor-stage-1-entrypoint
```

## Цель этапа

Подготовить безопасный каркас конструктора без полноценного расчёта признаков:

1. добавить кнопку **CONSTR** на Well Log вкладку кластерного анализа;
2. подключить открытие пустой/заглушечной формы конструктора;
3. подготовить файловую структуру будущего backend-сервиса;
4. зафиксировать основные правила и структуру конфигурации признака;
5. не менять поведение `COLLECT`.

## Объём работ

### 1.1. Добавление кнопки `CONSTR`

В UI Well Log вкладки кластерного анализа добавить кнопку:

```text
CONSTR
```

Рекомендуемый объектный name:

```text
pushButton_cluster_well_log_constructor
```

Требования:

- кнопка должна быть видна только в контексте Well Log dataset;
- если текущий dataset не выбран, при нажатии показывать предупреждение;
- если dataset выбран, открывать форму конструктора;
- кнопка не должна запускать расчёт напрямую;
- кнопка не должна менять состав dataset сама по себе.

### 1.2. Создание формы конструктора-заглушки

Создать новую форму, например:

```text
qt/well_log_feature_constructor_form.ui
qt/well_log_feature_constructor_form.py
```

Минимальная структура формы на этапе 1:

- заголовок текущего dataset;
- текстовое описание, что функциональность находится в разработке;
- список глобальных расчётных признаков, если они уже есть в БД;
- кнопка закрытия;
- кнопки `Validate`, `Save`, `Add to dataset` пока могут быть disabled или отсутствовать.

### 1.3. Создание backend-модуля-заготовки

Создать модуль:

```text
cluster/well_feature_calculator.py
```

В модуле подготовить:

- dataclass/структуру ошибки расчёта;
- dataclass/структуру конфигурации признака;
- функцию разбора `params_json`;
- функцию-заглушку валидации;
- функцию-заглушку расчёта.

Пример будущих публичных функций:

```python
parse_feature_calculator_config(calculator)
validate_feature_calculator_config(config)
validate_feature_calculator_for_dataset(dataset_id, calculator_id)
evaluate_feature_calculator_for_well(config, well_id, top_md, bottom_md)
```

На этом этапе функции могут возвращать контролируемую ошибку `NotImplemented`, но не должны падать неожиданно.

### 1.4. Формат `params_json`

Зафиксировать в коде и документации версию конфигурации:

```json
{
  "version": 1,
  "mode": "operation",
  "operation": "zscore",
  "inputs": [
    {
      "canonical_id": 12,
      "canonical_name": "GR"
    }
  ],
  "normalization_scope": "whole_well",
  "outlier_policy": "none"
}
```

Для formula mode:

```json
{
  "version": 1,
  "mode": "formula",
  "expression": "(GR - RHOB) / (GR + RHOB)",
  "inputs": [
    {
      "canonical_id": 12,
      "canonical_name": "GR"
    },
    {
      "canonical_id": 17,
      "canonical_name": "RHOB"
    }
  ],
  "depth_grid_policy": "strict_equal",
  "invalid_math_policy": "block",
  "outlier_policy": "none"
}
```

### 1.5. Навигация и состояние формы

При открытии конструктора форма должна получать:

- `dataset_id`;
- имя dataset;
- список скважин dataset;
- список canonical-параметров dataset;
- список уже привязанных calculator-параметров.

Если dataset пустой:

- форма может открываться;
- создание глобальных формул разрешено;
- добавление в dataset должно быть недоступно или сопровождаться предупреждением.

## Проверки этапа

1. Кнопка `CONSTR` отображается на Well Log вкладке.
2. Нажатие без выбранного dataset показывает предупреждение.
3. Нажатие с выбранным dataset открывает форму.
4. Форма показывает текущий dataset.
5. Приложение запускается без ошибок импорта.
6. Существующий `COLLECT` работает как раньше.

## Что не входит в этап

- расчёт операций;
- сохранение новых признаков;
- добавление признаков в dataset;
- формульный parser;
- preview;
- изменение `COLLECT`.

## Результат этапа

После этапа 1 пользователь видит кнопку `CONSTR` и может открыть отдельное окно конструктора, но полноценный расчёт ещё не доступен.

# Этап 2. Backend-ядро калькулятора: операции над одной кривой

## Ветка

```text
feature/well-log-constructor-stage-2-unary-backend
```

## Цель этапа

Реализовать backend-ядро расчёта признаков для операций над одной canonical-кривой без UI-сохранения и без formula mode.

## Объём работ

### 2.1. Загрузка входной кривой

Реализовать функцию загрузки кривой по:

- `well_id`;
- `canonical_id` или `canonical_name`;
- `top_md`;
- `bottom_md`;
- `normalization_scope`.

Функция должна учитывать алиасы canonical-кривых:

1. найти aliases для canonical-кривой;
2. найти фактическую `WellLog.curve_name` в скважине;
3. выбрать подходящую кривую;
4. разобрать `curve_data`;
5. построить массив глубин через `begin + index * step`;
6. вернуть только точки нужной области для расчёта или всю кривую для статистики нормировки.

### 2.2. Структура результата кривой

Рекомендуемая структура:

```python
@dataclass
class WellLogCurveSeries:
    well_id: int
    canonical_id: int
    canonical_name: str
    depths: list[float]
    values: list[float | None]
    step: float
    begin: float
    end: float
    source_curve_name: str
```

### 2.3. Реализация унарных операций

Реализовать операции:

- `log`;
- `abs`;
- `sqrt`;
- `derivative`;
- `rolling_mean`;
- `rolling_median`;
- `zscore`;
- `robust`;
- `minmax`.

Требования к операциям:

- все операции должны быть pure functions;
- операции не должны обращаться к UI;
- операции не должны писать в БД;
- ошибки должны возвращаться структурированно;
- недопустимые значения должны приводить к блокирующей ошибке, если итоговый признак не может быть полностью рассчитан;
- деление на данном этапе не реализуется, так как это binary/formula функционал.

### 2.4. Нормировка внутри скважины

Для `zscore`, `robust`, `minmax` реализовать параметр:

```text
normalization_scope
```

Допустимые значения:

```text
whole_well
interval
```

Поведение:

- `whole_well`:
  - статистика считается по всей кривой в скважине;
  - в результат возвращаются значения только выбранного интервала;
- `interval`:
  - статистика считается только по выбранному интервалу;
  - результат также ограничен выбранным интервалом.

### 2.5. Обработка зашкал

В MVP использовать:

```text
outlier_policy = none
```

Но в конфигурации и dataclass сразу предусмотреть поле:

```python
outlier_policy: str = "none"
```

На этом этапе:

- значения не клипуются;
- выбросы не маскируются;
- если значения конечные, расчёт продолжается;
- если появляются `NaN`/`inf`, возвращается блокирующая ошибка.

### 2.6. Ошибки расчёта

Ввести единый тип ошибки, например:

```python
@dataclass
class FeatureCalculatorError:
    code: str
    message: str
    calculator_id: int | None = None
    feature_name: str | None = None
    well_id: int | None = None
    well_name: str | None = None
    canonical_name: str | None = None
    severity: str = "error"
```

Минимальные коды ошибок:

- `missing_curve`;
- `empty_curve_data`;
- `invalid_curve_json`;
- `invalid_step`;
- `not_enough_points`;
- `invalid_math_domain`;
- `zero_std`;
- `zero_iqr`;
- `zero_range`;
- `non_finite_result`;
- `unsupported_operation`.

## Тесты этапа

Добавить unit-тесты на pure functions:

1. `log` на положительных значениях;
2. `log` на нуле/отрицательных значениях;
3. `sqrt` на положительных и отрицательных значениях;
4. `derivative` на равномерной сетке;
5. `rolling_mean`;
6. `rolling_median`;
7. `zscore`;
8. `robust`;
9. `minmax`;
10. ошибки нулевого std/IQR/range;
11. ошибки `NaN`/`inf`.

## Что не входит в этап

- formula mode;
- операции между несколькими кривыми;
- UI создания признаков;
- интеграция расчётных признаков в `COLLECT`;
- preview.

## Результат этапа

После этапа 2 в проекте есть протестированное backend-ядро для унарных операций над одной кривой.

# Этап 3. Строгая проверка глубинных сеток и backend для формул

## Ветка

```text
feature/well-log-constructor-stage-3-formula-backend
```

## Цель этапа

Добавить backend-поддержку формул с несколькими входными кривыми и строгой проверкой, что кривые можно использовать вместе без интерполяции.

## Объём работ

### 3.1. Загрузка нескольких входных кривых

Реализовать функцию, которая для одной скважины загружает все входные canonical-кривые признака:

```python
load_input_series_for_calculator(config, well_id, top_md, bottom_md)
```

Функция должна вернуть:

- список `WellLogCurveSeries`;
- ошибки отсутствующих кривых;
- ошибки некорректных данных.

### 3.2. Проверка совместимости глубинной сетки

Реализовать строгую проверку:

1. все входные кривые имеют валидный `step`;
2. `step` совпадает в пределах допуска;
3. список глубин в расчётном интервале совпадает после округления до единого precision;
4. количество точек совпадает;
5. первая и последняя глубина совпадают;
6. если хотя бы одно условие нарушено, возвращается блокирующая ошибка.

Рекомендуемый код ошибки:

```text
depth_grid_mismatch
```

Сообщение должно содержать:

- имя признака;
- скважину;
- входные кривые;
- различающиеся шаги или диапазоны;
- пояснение, что интерполяция в текущей версии не используется.

### 3.3. Безопасный parser формул

Реализовать parser на основе `ast`, а не через прямой `eval`.

Разрешить:

- имена входных кривых;
- числовые константы;
- скобки;
- операторы `+`, `-`, `*`, `/`;
- unary `+` и unary `-`;
- функции:
  - `log`;
  - `abs`;
  - `sqrt`.

Запретить:

- любые импорты;
- атрибуты;
- индексацию;
- lambda;
- comprehensions;
- вызовы неизвестных функций;
- доступ к builtins;
- имена, которых нет в `inputs`.

### 3.4. Расчёт формулы

Для каждой глубинной точки:

1. подготовить словарь значений входных кривых;
2. вычислить выражение через безопасный evaluator;
3. проверить математическую область;
4. проверить конечность результата;
5. добавить значение в результат.

При любой точечной ошибке в MVP возвращать блокирующую ошибку для всего признака. Политики частичных пропусков не реализуются.

### 3.5. Арифметические ошибки

Обработать:

- деление на ноль;
- `log(x)` при `x <= 0`;
- `sqrt(x)` при `x < 0`;
- `NaN`;
- `inf`;
- отсутствие значения входной кривой на глубине.

Коды ошибок:

- `division_by_zero`;
- `invalid_log_domain`;
- `invalid_sqrt_domain`;
- `missing_value_at_depth`;
- `non_finite_result`;
- `formula_parse_error`;
- `formula_validation_error`.

## Тесты этапа

1. успешная формула `A + B`;
2. успешная формула `(A - B) / (A + B)`;
3. ошибка деления на ноль;
4. ошибка `log` от отрицательного значения;
5. ошибка неизвестной кривой;
6. ошибка неизвестной функции;
7. ошибка попытки доступа к атрибуту;
8. ошибка несовпадения `step`;
9. ошибка несовпадения глубинных точек при одинаковом `step`;
10. успешный расчёт при полностью совпадающей сетке.

## Что не входит в этап

- UI formula editor;
- сохранение признаков через форму;
- интеграция в `COLLECT`;
- preview;
- интерполяция.

## Результат этапа

После этапа 3 backend умеет безопасно считать формулы по нескольким кривым при условии строгого совпадения глубинных сеток.

# Этап 4. Интеграция расчётных признаков в `COLLECT`

## Ветка

```text
feature/well-log-constructor-stage-4-collect-integration
```

## Цель этапа

Сделать так, чтобы выбранные calculator-признаки реально попадали в итоговый `data` текущего Well Log dataset при нажатии `COLLECT`.

## Объём работ

### 4.1. Загрузка calculator-параметров dataset

В `collect_cluster_well_log_dataset_data()` добавить загрузку:

```python
ClusterWellLogParameterFromCalculator
```

с join на:

```python
FeatureCalculator
```

Порядок колонок должен быть стабильным:

1. `well_id_depth`;
2. sample index;
3. canonical-признаки по имени;
4. calculator-признаки по `feature_name`.

### 4.2. Preflight validation перед сборкой

Перед созданием итоговых строк выполнить полную проверку:

```python
validate_calculators_for_dataset(dataset_id)
```

Проверка должна пройти по всем:

- выбранным calculator-признакам;
- скважинам dataset;
- выбранным интервалам скважин.

Если найдена хотя бы одна ошибка:

- `COLLECT` не выполняется;
- старое `data` не удаляется;
- кнопка `COLLECT` не переводится в зелёное состояние;
- пользователь получает подробное сообщение;
- ошибки желательно ограничить в диалоге первыми N строками и предложить смотреть полный лог в `set_info`/логовом окне.

### 4.3. Расчёт calculator-признаков по скважине

Для каждой скважины:

1. собрать canonical `depth_map`, как сейчас;
2. отдельно рассчитать calculator features через backend-сервис;
3. объединить значения по глубинам;
4. сформировать строки итоговой таблицы.

Важно:

- глубинные точки calculator-признака должны соответствовать строкам итогового dataset;
- если calculator-признак рассчитан на глубинах, которых нет в canonical `depth_map`, нужно решить поведение до реализации.

Для MVP рекомендуемое правило:

```text
итоговая сетка строк формируется как union глубин canonical-параметров и calculator-признаков, но каждый выбранный признак должен иметь значение на каждой своей допустимой глубине; отсутствие значения выбранного признака на строке итоговой сетки блокирует сборку.
```

Альтернативно, более строгое правило:

```text
calculator-признак должен возвращать значения на той же сетке, что и его входные кривые; итоговый dataset строится по union всех доступных глубин, но строки с неполным набором выбранных признаков блокируют сборку.
```

В MVP лучше использовать строгий вариант, чтобы не получить скрытые пропуски.

### 4.4. Инвалидация `data`

При добавлении/удалении calculator-признака из dataset:

- удалять текущую запись `WellLogClusterDatasetData`;
- переводить кнопку `COLLECT` в состояние “не актуально”;
- обновлять список параметров.

Это должно соответствовать уже принятому правилу: любое изменение входных параметров dataset инвалидирует собранные данные.

### 4.5. Сообщения пользователю

Успешный сбор должен сообщать:

- количество строк;
- количество canonical-признаков;
- количество calculator-признаков;
- общее количество признаков.

Пример:

```text
COLLECT WELL LOG: собрано строк 15234, canonical признаков 4, calculator признаков 2, всего признаков 7 включая sample_index.
```

Ошибка должна сообщать:

```text
COLLECT WELL LOG заблокирован: 3 ошибки расчёта признаков.
1) GR_RHOB_RATIO / Well-12: несовместимая глубинная сетка GR и RHOB.
2) LOG_GR / Well-15: log недопустим для значения <= 0 на глубине 1450.5.
```

## Тесты этапа

1. dataset с одним calculator-признаком успешно собирается;
2. в итоговом `data` есть колонка calculator-признака;
3. canonical и calculator колонки идут в стабильном порядке;
4. ошибка calculator-признака блокирует `COLLECT`;
5. при ошибке старый `data` не удаляется;
6. добавление calculator-признака инвалидирует старый `data`;
7. удаление calculator-признака инвалидирует старый `data`;
8. dataset без calculator-признаков собирается как раньше.

## Что не входит в этап

- полноценный UI создания признака;
- preview;
- редактирование признака;
- интерполяция;
- частичные пропуски.

## Результат этапа

После этапа 4 расчётные признаки, уже существующие в БД и привязанные к dataset, полноценно участвуют в `COLLECT`.

# Этап 5. UI создания operation-признаков

## Ветка

```text
feature/well-log-constructor-stage-5-operation-ui
```

## Цель этапа

Дать пользователю возможность создавать расчётные признаки через форму конструктора без ручной работы с БД. На этом этапе реализуются только operation-признаки, то есть операции над одной кривой.

## Объём работ

### 5.1. Расширение формы конструктора

Форма должна содержать:

1. блок текущего dataset;
2. список доступных canonical-кривых;
3. список глобальных calculator-признаков;
4. блок создания нового признака;
5. блок действий с выбранным признаком.

### 5.2. Поля создания operation-признака

Минимальный набор полей:

```text
Feature name
Mode: Operation
Input curve
Operation
Normalization scope
Rolling window
Min periods
Outlier policy
```

Поле `Rolling window` показывать или активировать только для:

- `rolling_mean`;
- `rolling_median`.

Поле `Normalization scope` показывать или активировать только для:

- `zscore`;
- `robust`;
- `minmax`.

### 5.3. Валидация формы

Перед сохранением проверять:

- имя не пустое;
- имя уникальное среди `FeatureCalculator.feature_name`;
- имя не совпадает с canonical-именем;
- выбран input curve;
- выбран operation;
- параметры operation корректны;
- `normalization_scope` задан для нормировок;
- `window > 0` для rolling операций;
- `min_periods > 0` и `min_periods <= window`.

### 5.4. Сохранение признака

При `SAVE` создать запись:

```text
FeatureCalculator
```

Заполнить:

- `feature_name`;
- `expression` — человекочитаемое представление, например `zscore(GR)`;
- `used_canonical_well_log` — JSON со списком canonical names/ids;
- `transform_type` — operation name;
- `params_json` — полный JSON версии 1.

После сохранения:

- обновить список глобальных признаков;
- выбрать новый признак в списке;
- показать сообщение об успешном сохранении.

### 5.5. Добавление признака в текущий dataset

Кнопка:

```text
ADD TO DATASET
```

Поведение:

1. проверить, что выбран текущий dataset;
2. проверить, что выбран calculator-признак;
3. выполнить validation применимости к dataset;
4. если есть ошибки — не добавлять;
5. если ошибок нет — создать `ClusterWellLogParameterFromCalculator`;
6. если такая связь уже есть — показать информационное сообщение;
7. инвалидировать `data` текущего dataset;
8. обновить `listWidget_cluster_list_log` на основной вкладке.

### 5.6. Удаление признака из текущего dataset

На форме конструктора добавить кнопку:

```text
REMOVE FROM DATASET
```

Поведение:

- удаляет только связь признака с текущим dataset;
- не удаляет глобальный `FeatureCalculator`;
- инвалидирует `data`;
- обновляет основной список параметров.

### 5.7. Просмотр определения признака

При выборе признака из глобальной библиотеки показывать:

- имя;
- тип;
- expression;
- inputs;
- normalization scope;
- outlier policy;
- дату создания;
- используется ли в текущем dataset.

## Тесты этапа

1. создание `log(GR)`;
2. создание `sqrt(GR)`;
3. создание `rolling_mean(GR, 5)`;
4. создание `zscore(GR)` с `whole_well`;
5. создание `zscore(GR)` с `interval`;
6. запрет пустого имени;
7. запрет duplicate имени;
8. запрет rolling window <= 0;
9. добавление признака в dataset;
10. повторное добавление не создаёт дубль;
11. удаление признака из dataset не удаляет глобальный признак;
12. добавление/удаление инвалидирует `data`.

## Что не входит в этап

- formula UI;
- preview-график;
- edit существующих признаков;
- удаление глобальных признаков;
- интерполяция.

## Результат этапа

После этапа 5 пользователь может создавать operation-признаки через форму и добавлять их в текущий Well Log dataset.

# Общие требования ко всем этапам

## 1. Ветки и pull requests

Каждый этап выполняется в отдельной ветке:

```text
feature/well-log-constructor-stage-N-...
```

Каждый PR должен содержать:

- краткую цель этапа;
- список изменений;
- список затронутых файлов;
- результаты тестов;
- известные ограничения;
- скриншот, если менялся UI;
- подтверждение, что предыдущий workflow Well Log кластеризации не сломан.

## 2. Совместимость

На каждом этапе необходимо проверять:

- приложение запускается;
- существующий Well Log dataset можно создать/выбрать;
- обычные canonical-параметры добавляются как раньше;
- `COLLECT` без calculator-признаков работает как раньше;
- удаление параметров инвалидирует `data` как раньше.

## 3. Принцип строгой безопасности расчёта

До появления явных политик пропусков и интерполяции действует правило:

```text
Если выбранный расчётный признак не может быть полностью и однозначно рассчитан — блокируем добавление в dataset или COLLECT.
```

## 4. Принцип отсутствия скрытой интерполяции

Запрещено автоматически интерполировать кривые в MVP. Если глубинные сетки не совпадают, пользователь должен получить понятную ошибку.

## 5. Принцип глобальности признаков

`FeatureCalculator` — глобальная библиотека признаков. Привязка к dataset выполняется отдельно через связь calculator-priznak ↔ dataset.

## 6. Принцип неизменяемости признаков в MVP

Существующие признаки не редактируются in-place. Если нужно изменить формулу, пользователь создаёт новый признак.

## 7. Принцип проверяемости

Backend-операции должны быть покрыты unit-тестами. UI-часть должна иметь хотя бы smoke/integration проверки или подробные ручные тест-кейсы в PR.

## 8. Принцип понятных ошибок

Любая ошибка расчёта должна быть понятна пользователю и содержать:

- имя признака;
- имя/ID скважины;
- входные кривые;
- причину;
- рекомендацию.

# Итоговый целевой пользовательский сценарий

1. Пользователь создаёт или выбирает Well Log dataset.
2. Добавляет скважины.
3. Выбирает интервалы.
4. Добавляет обычные canonical-параметры.
5. Нажимает **CONSTR**.
6. В открывшемся конструкторе создаёт расчётный признак или выбирает существующий глобальный признак.
7. Проверяет coverage и preview.
8. Нажимает `ADD TO DATASET`.
9. Конструктор валидирует признак для текущего dataset.
10. Если ошибок нет, признак появляется в списке параметров как `[calc]`.
11. Кнопка `COLLECT` становится неактуальной до пересборки.
12. Пользователь нажимает `COLLECT`.
13. Dataset собирается с canonical и calculator-признаками.
14. Если хотя бы один расчётный признак не считается, `COLLECT` блокируется с подробной ошибкой.
15. После успешного `COLLECT` пользователь запускает дальнейший анализ/кластеризацию.
