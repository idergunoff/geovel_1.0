# Поэтапный план реализации расчёта, анализа и визуализации кластеризации для вкладки Well Log

## 1. Цель этапа

Реализовать для вкладки скважинного каротажа (`Well Log`) полный рабочий цикл кластерного анализа:

1. запуск расчёта по кнопке `CALC` с выбором источника данных по активной вкладке;
2. автоматический подбор параметров по аналогии с уже реализуемым/запланированным автоподбором для георадара;
3. сохранение и отображение лучших вариантов расчёта в общей таблице результатов автоподбора;
4. автоматическое обновление таблицы результатов при переключении между вкладками `Георадар` / `Well Log`;
5. переиспользование графиков качества кластеризации из георадарного расчёта;
6. отдельное окно визуализации результата для скважинного каротажа с тремя обязательными режимами:
   - **одна скважина → все кривые**;
   - **одна кривая → разные скважины**;
   - **средний портрет кластера**.

---

## 2. Что считаем базой и что переиспользуем

### 2.1. Уже существующая логика подготовки Well Log dataset

Вкладка Well Log должна опираться на ранее описанный процесс подготовки набора:

- создание/выбор `WellLogClusterDataset`;
- добавление скважин в набор;
- выбор интервалов по скважинам;
- добавление канонических каротажных параметров;
- сбор итоговых строк в `WellLogClusterDatasetData.data` через `COLLECT`.

Для текущего этапа считаем, что `CALC` и `AUTO` не должны сами собирать датасет заново. Они работают только с уже собранным и актуальным `data`. Если данных нет или набор неактуален, пользователь должен получить блокирующее сообщение с предложением выполнить `COLLECT`.

### 2.2. Уже существующая логика георадарной кластеризации

Из текущего пайплайна кластеризации нужно переиспользовать, а не дублировать:

- очистку признаков;
- масштабирование;
- PCA;
- запуск алгоритмов кластеризации;
- расчёт метрик качества;
- автоподбор `COARSE` / `FINE`;
- таблицу лучших вариантов;
- графики качества расчёта.

Цель — сделать общий backend для расчёта, где различается только источник данных и metadata строк.

---

## 3. Единая модель контекста расчёта

### 3.1. Активная вкладка как источник данных

При нажатии `CALC`, `AUTO` и при перерисовке таблицы результатов нужно определять активный контекст:

```text
active_cluster_context = "gpr" | "well_log"
```

Рекомендуемая логика:

1. если активна вкладка георадара:
   - источник данных: текущий `ObjectSet.data`;
   - metadata строк: профиль, трасса, координата/индекс профиля;
   - тип результата: `gpr_cluster_result`.
2. если активна вкладка Well Log:
   - источник данных: текущий `WellLogClusterDatasetData.data`;
   - metadata строк: `well_id`, `well_name`, `depth_md`, исходные кривые;
   - тип результата: `well_log_cluster_result`.

### 3.2. Единый объект контекста

Добавить внутренний объект/словарь `ClusterRunContext`:

```python
ClusterRunContext = {
    "source_type": "gpr" | "well_log",
    "dataset_id": int,
    "dataset_title": str,
    "raw_rows": list,
    "feature_names": list[str],
    "meta_columns": list[str],
    "feature_columns": list[str],
    "row_count": int,
    "ui_tab_key": str,
}
```

Для Well Log обязательные metadata:

```text
well_id
well_name
depth_md
row_index_in_well
```

`row_index_in_well` нужен для стабильной отрисовки последовательности по глубине и для сглаживания/агрегации интервалов.

### 3.3. Критерий готовности

- `CALC` и `AUTO` получают данные только через `build_cluster_run_context()`.
- В коде нет отдельных несвязанных веток с дублированием всего пайплайна.
- При отсутствии данных для активной вкладки показывается понятное сообщение:
  - для георадара: выбрать/собрать объектный набор;
  - для Well Log: выбрать dataset и выполнить `COLLECT`.

## 4. Формат данных Well Log для расчёта

### 4.1. Входной формат после `COLLECT`

Для Well Log каждая строка должна соответствовать одной глубинной точке одной скважины:

```json
{
  "well_id": 12,
  "well_name": "W-12",
  "depth_md": 1543.2,
  "features": {
    "GR": 83.1,
    "RHOB": 2.42,
    "NPHI": 0.18
  },
  "source_curve_names": {
    "GR": "GK",
    "RHOB": "RHOB",
    "NPHI": "NPHI"
  }
}
```

Если текущий `data` хранится компактнее, на уровне `build_well_log_cluster_context()` нужно привести его к единому runtime-формату:

```text
X: numeric matrix, shape = n_rows × n_features
meta: list[dict], length = n_rows
feature_names: list[str]
```

### 4.2. Политика пропусков

На этапе расчёта для Well Log использовать те же настройки, что и для георадара, но добавить well-specific проверки:

1. строка без `well_id` или `depth_md` не допускается;
2. скважина с количеством валидных строк ниже порога исключается из расчёта с предупреждением;
3. если после очистки осталось меньше двух скважин или слишком мало строк, расчёт блокируется;
4. все исключённые строки/скважины должны попадать в diagnostic summary.

### 4.3. Критерий готовности

- На один и тот же Well Log dataset можно повторно запускать `CALC` без пересборки `COLLECT`.
- Результат содержит связь каждой метки кластера с `well_id` и `depth_md`.

## 5. Ручной расчёт по кнопке `CALC`

### 5.1. Поведение кнопки

При нажатии `CALC`:

1. определить активную вкладку;
2. построить `ClusterRunContext`;
3. провалидировать наличие данных;
4. прочитать текущие параметры UI:
   - scaler;
   - PCA;
   - метод кластеризации;
   - параметры метода;
   - параметры сглаживания, если включены;
5. выполнить общий pipeline кластеризации;
6. сохранить результат расчёта в кэш/таблицу результатов активного dataset;
7. обновить графики качества;
8. открыть или обновить окно визуализации Well Log, если активна вкладка Well Log.

### 5.2. Что сохранять после расчёта

Для каждого ручного запуска сохранять:

```text
run_id
source_type = well_log
dataset_id
created_at
config_json
metrics_json
labels_json или ссылка на labels blob
meta_snapshot_hash
feature_names_json
status
error_text
```

Для MVP можно хранить только последний успешный ручной результат в памяти текущей сессии, но архитектурно лучше сразу заложить отдельное хранение, потому что визуализация должна переживать переключение вкладок и повторное открытие окна.

### 5.3. Критерий готовности

- `CALC` на вкладке Well Log считает кластеры по выбранному dataset.
- После расчёта доступны метрики качества и labels для визуализации.
- Георадарный `CALC` не ломается и продолжает работать на своих данных.

---

## 6. Автоподбор параметров для Well Log

### 6.1. Общий принцип

Автоподбор для Well Log должен работать по той же схеме, что и для георадара:

1. `COARSE` — быстрый широкий поиск;
2. `FINE` — локальный поиск вокруг лучших кандидатов;
3. единый scoring;
4. сохранение leaderboard;
5. возможность применить лучший вариант к UI и пересчитать/отрисовать результат.

### 6.2. Well-specific ограничения search space

Для скважинного каротажа нужно добавить ограничения, чтобы автоподбор не выбирал визуально бесполезные решения:

1. минимальное число кластеров: `2`;
2. максимальное число кластеров по умолчанию: `8` или `min(12, sqrt(n_rows))`;
3. минимальная доля строк в непустом кластере: например `1–2%`;
4. максимальная доля noise для HDBSCAN: например `40%`;
5. штраф за слишком раздробленную разметку по глубине;
6. штраф за кластеры, встречающиеся только в одной скважине, если выбран режим межскважинного анализа.

### 6.3. Candidate config

Использовать тот же формат `candidate_config`, что и для георадара:

```json
{
  "scaler_mode": "standard",
  "pca_enabled": true,
  "pca_mode": "variance_ratio",
  "pca_value": 0.9,
  "method": "kmeans",
  "method_params": {
    "n_clusters": 4,
    "random_state": 42
  }
}
```

### 6.4. Дополнительные метрики для Well Log

Помимо общих метрик:

- Silhouette;
- Davies-Bouldin;
- Calinski-Harabasz;
- число кластеров;
- доля noise;

добавить well-specific diagnostics:

1. `mean_segments_per_well` — среднее число интервалов кластеров на скважину;
2. `median_segment_thickness` — медианная мощность непрерывного интервала кластера;
3. `cluster_well_coverage` — в скольких скважинах встречается каждый кластер;
4. `dominant_cluster_balance` — насколько один кластер не доминирует во всём наборе;
5. `depth_fragmentation_penalty` — штраф за частую смену кластеров по глубине.

Эти метрики не обязательно сразу выводить все в UI, но их нужно сохранить в `diagnostics_json` для последующего анализа.

### 6.5. Scoring

Итоговый score для Well Log:

```text
score = common_quality_score
        - noise_penalty
        - tiny_cluster_penalty
        - depth_fragmentation_penalty
        + well_coverage_bonus
```

Где:

- `common_quality_score` считается по общей логике георадара;
- `noise_penalty` актуален для HDBSCAN;
- `tiny_cluster_penalty` штрафует кластеры с очень малым количеством точек;
- `depth_fragmentation_penalty` штрафует частое чередование меток на соседних глубинах;
- `well_coverage_bonus` поощряет кластеры, встречающиеся в нескольких скважинах.

### 6.6. Сохранение лучших вариантов

Лучшие варианты должны сохраняться в таблицу результатов с привязкой к активному источнику:

```text
source_type
dataset_id
rank
score
method
scaler
pca
silhouette
davies_bouldin
calinski_harabasz
n_clusters
noise_fraction
well_log_diagnostics_json
candidate_config_json
created_at
```

Для UI таблицы достаточно отображать компактный набор колонок:

| Rank | Score | Source | Dataset | Method | Scaler | PCA | Silhouette | DB | CH | Clusters | Noise % | Well diagnostics | Status |
|---:|---:|---|---|---|---|---|---:|---:|---:|---:|---:|---|---|

`Well diagnostics` в MVP можно сделать краткой строкой, например:

```text
segments=4.2/well, med_thick=1.6m, coverage=5/6 wells
```

Полный JSON конфигурации и диагностик — в tooltip строки.

### 6.7. Критерий готовности

- `AUTO` на вкладке Well Log запускает `COARSE` / `FINE` и заполняет таблицу результатов.
- Лучший вариант можно применить к UI-контролам.
- При двойном клике по строке leaderboard можно открыть визуализацию результата данного кандидата.
- Георадарный автоподбор продолжает использовать ту же таблицу, но фильтруется по `source_type = gpr`.

## 7. Обновление таблицы результатов при переключении вкладок

### 7.1. Требуемое поведение

Таблица лучших вариантов должна быть общей в интерфейсе, но её содержимое зависит от активной вкладки.

При переключении вкладки:

1. определить `active_cluster_context`;
2. получить текущий `dataset_id` для этой вкладки;
3. очистить `tableWidget_cluster_auto_result`;
4. загрузить сохранённые результаты только для пары:

```text
(source_type, dataset_id)
```

5. восстановить выделение последнего выбранного результата, если он всё ещё существует;
6. обновить графики качества под выбранную строку.

### 7.2. События, которые должны обновлять таблицу

Таблица обновляется при:

- переключении вкладок `Георадар` / `Well Log`;
- смене Well Log dataset в `comboBox_cluster_well_set`;
- смене георадарного object set;
- завершении `AUTO`;
- завершении ручного `CALC`, если ручной результат тоже добавляем в список;
- удалении dataset;
- пересборке `COLLECT`, если старые результаты инвалидируются.

### 7.3. Инвалидация результатов

Если Well Log dataset был пересобран через `COLLECT`, старые результаты нельзя молча показывать как актуальные.

Рекомендуемый механизм:

1. при `COLLECT` считать `data_hash`;
2. сохранять `data_hash` в каждом результате автоподбора/расчёта;
3. при загрузке таблицы сравнивать текущий `data_hash` с сохранённым;
4. если hash отличается:
   - либо скрывать результат;
   - либо показывать статус `stale` серым цветом;
   - запуск визуализации для stale-результата запрещать или требовать подтверждение.

### 7.4. Критерий готовности

- Переключение вкладок мгновенно меняет содержимое таблицы результатов.
- Результаты георадара и Well Log не смешиваются.
- После пересборки Well Log dataset старые результаты отмечаются как неактуальные.

## 8. Графики качества кластеризации

### 8.1. Что переиспользуем

Для Well Log переиспользовать те же графики, что уже используются/планируются для георадара:

1. bar/line chart по `score` для top-N кандидатов;
2. сравнение Silhouette / DB / CH;
3. распределение размеров кластеров;
4. доля noise;
5. PCA scatter или 2D projection, если доступна;
6. summary выбранного candidate config.

### 8.2. Что добавить для Well Log

Добавить отдельный блок diagnostics:

1. распределение длины непрерывных интервалов кластеров;
2. число смен кластеров по каждой скважине;
3. покрытие кластеров по скважинам;
4. stacked bar: доля кластеров в каждой скважине.

Это можно сделать вторым этапом после MVP. Для первого этапа достаточно переиспользовать общие графики качества и добавить `cluster size distribution`.

### 8.3. Критерий готовности

- При выборе строки в таблице результатов графики качества обновляются.
- Один и тот же компонент графиков умеет принимать `source_type = gpr` и `source_type = well_log`.
- Для Well Log минимум отображаются общие метрики и размеры кластеров.

## 9. Окно визуализации Well Log результата

### 9.1. Общая концепция окна

Создать отдельное окно, например:

```text
WellLogClusterResultWindow
```

Оно открывается:

- автоматически после успешного `CALC` на вкладке Well Log;
- по кнопке `SHOW RESULT`;
- по двойному клику на строке таблицы лучших вариантов.

Окно получает:

```python
{
    "dataset_id": int,
    "run_id": int,
    "feature_names": list[str],
    "labels": list[int],
    "meta": list[dict],
    "X_clean": np.ndarray,
    "cluster_profiles": dict,
    "metrics": dict,
    "config": dict,
}
```

### 9.2. Основные элементы окна

Верхняя панель:

- dataset name;
- method + key parameters;
- metrics summary;
- selector `View mode`:
  - `Well: all curves`;
  - `Curve: all wells`;
  - `Cluster profile`.

Левая панель:

- список/combobox скважин;
- список/combobox кривых;
- фильтр кластеров;
- переключатель `show noise`;
- настройка `cluster opacity`;
- кнопка export screenshot/table.

Центральная область:

- matplotlib/pyqtgraph canvas с выбранной визуализацией.

Нижняя область:

- легенда кластеров;
- таблица интервалов выбранной скважины или выбранной кривой.

## 10. Визуализация 1: одна скважина → все кривые

### 10.1. Цель

Показать, как кластеры распределяются по глубине внутри одной скважины и как границы кластеров соотносятся со всеми выбранными каротажными кривыми.

### 10.2. Layout

Для выбранной скважины:

```text
Depth ↓

[GR curve] [RHOB curve] [NPHI curve] [Cluster track]
```

Обязательные элементы:

1. общая ось глубины для всех треков;
2. инвертированная ось Y: глубина растёт вниз;
3. отдельный `Cluster track` справа или слева;
4. одинаковая палитра кластеров во всех режимах;
5. подписи кривых и единиц, если доступны;
6. tooltip/статусная строка:
   - скважина;
   - глубина;
   - значение кривой;
   - cluster id.

### 10.3. Отрисовка кластера

Основной вариант — не красить сами линии, а рисовать отдельную колонку интервалов:

```text
Cluster track:
1540.0–1542.4 → cluster 1
1542.4–1547.0 → cluster 3
1547.0–1551.2 → cluster 2
```

Дополнительно можно включить `background mode`, где цвет кластера отображается полупрозрачным фоном на всех треках.

### 10.4. Сглаживание для отображения

Визуальное сглаживание должно быть отдельным от исходных labels:

- исходные labels сохраняются без изменений;
- для display можно объединять короткие интервалы меньше заданной мощности;
- пользователь должен видеть, включено ли display-smoothing.

### 10.5. Таблица интервалов

Под графиком показывать таблицу:

| Well | From MD | To MD | Thickness | Cluster | Rows | Dominant curves summary |
|---|---:|---:|---:|---:|---:|---|

Клик по строке подсвечивает интервал на графике.

### 10.6. Критерий готовности

- Можно выбрать скважину и увидеть все кривые с cluster track.
- При смене скважины график перерисовывается без повторного расчёта.
- Цвета кластеров совпадают с легендой и другими режимами.

## 11. Визуализация 2: одна кривая → разные скважины

### 11.1. Цель

Сравнить поведение одной и той же каротажной кривой и кластерной разметки между несколькими скважинами.

### 11.2. Layout

Для выбранной кривой:

```text
Curve: GR

[Well A: GR + cluster background]
[Well B: GR + cluster background]
[Well C: GR + cluster background]
```

Возможны два режима расположения:

1. `small multiples` — каждая скважина отдельным вертикальным треком;
2. `overlay normalized` — кривые разных скважин наложены после нормализации.

Для MVP лучше выбрать `small multiples`, потому что он проще и лучше работает при разных интервалах глубин.

### 11.3. Управление

Обязательные контролы:

- выбор кривой;
- выбор списка скважин;
- сортировка скважин:
  - по имени;
  - по доле выбранного кластера;
  - по похожести последовательности кластеров;
- ограничение количества скважин на экране, например top 12;
- переключатель общей/локальной шкалы X.

### 11.4. Отображение кластеров

Для каждой скважины:

- линия выбранной кривой;
- цветной фон или тонкая cluster track колонка;
- подпись скважины;
- мини-статистика: доля кластеров в интервале.

### 11.5. Критерий готовности

- Можно выбрать кривую и увидеть её по нескольким скважинам.
- Кластеры отображаются одинаковой палитрой.
- Сравнение не требует повторного `CALC`.

## 12. Визуализация 3: средний портрет кластера

### 12.1. Цель

Сделать кластеры интерпретируемыми: пользователь должен понимать не только `cluster 0`, `cluster 1`, `cluster 2`, а чем они отличаются по выбранным каротажным признакам.

### 12.2. Что считать

Для каждого кластера и каждого признака:

```text
count
mean
median
std
p10
p25
p75
p90
min
max
```

Дополнительно:

```text
cluster_size
cluster_fraction
wells_count
wells_fraction
mean_depth_md
min_depth_md
max_depth_md
```

Noise `-1` показывать отдельно и по умолчанию можно выключить.

### 12.3. Основные графики

MVP-набор:

1. **bar chart средних standardized значений** по признакам для каждого кластера;
2. **heatmap cluster × feature** по средним или медианам;
3. **boxplot/violin** для выбранного признака по кластерам;
4. **таблица профилей кластеров**.

Рекомендуемый основной вид:

```text
Cluster profile heatmap
Rows: cluster id
Columns: GR, RHOB, NPHI, ...
Value: z-score mean или median
```

Почему z-score:

- разные кривые имеют разные единицы и масштабы;
- так сразу видно, какие признаки выше/ниже среднего в каждом кластере.

### 12.4. Подписи интерпретации

Автоматически формировать краткое текстовое описание:

```text
Cluster 2: высокий GR, низкий RHOB, повышенный NPHI, встречается в 5/7 скважинах, 18% строк.
```

Правило:

- брать признаки, где `abs(z_mean) >= threshold`, например `0.7`;
- сортировать по абсолютному отклонению;
- выводить top-3 признака.

### 12.5. Критерий готовности

- После расчёта доступен средний портрет каждого кластера.
- Пользователь видит таблицу и минимум один график профиля.
- Цвет кластера в профиле совпадает с цветом в треках скважин.

ClusterInterval = {

## 17. Риски и решения

### 17.1. Большой объём строк Well Log

**Риск:** автоподбор будет медленным.

**Решение:** использовать sampling для AUTO, но финальный `CALC` лучшего candidate выполнять на полном dataset.

### 17.2. Слишком частая смена кластеров по глубине

**Риск:** результат формально хороший по Silhouette, но геологически нечитабельный.

**Решение:** добавить `depth_fragmentation_penalty` и display-smoothing.

### 17.3. Разные глубинные интервалы скважин

**Риск:** сравнение кривой между скважинами может быть визуально некорректным.

**Решение:** для MVP использовать small multiples с индивидуальной глубинной осью; позже добавить нормализацию интервала 0–100%.

### 17.4. Смешивание результатов георадара и Well Log

**Риск:** пользователь применит candidate не к тому dataset.

**Решение:** в каждом результате хранить `source_type`, `dataset_id`, `data_hash`; UI всегда фильтрует таблицу по активному контексту.

## 18. Контрольные тест-кейсы

1. `CALC` на Well Log без выбранного dataset → блокирующее сообщение.
2. `CALC` на Well Log с dataset без `COLLECT data` → блокирующее сообщение.
3. `CALC` на Well Log с валидным data → labels, metrics, result object.
4. `AUTO COARSE` на Well Log → top-N строк в таблице.
5. `AUTO FINE` на Well Log → fine-result не хуже coarse-best по score или фиксируется причина.
6. Переключение `Георадар → Well Log` → таблица показывает только Well Log результаты.
7. Переключение `Well Log → Георадар` → таблица показывает только георадарные результаты.
8. Смена `comboBox_cluster_well_set` → таблица перезагружается для нового dataset.
9. Пересборка `COLLECT` → старые результаты становятся `stale`.
10. Двойной клик по строке Well Log leaderboard → открывается окно визуализации.
11. В режиме `одна скважина → все кривые` смена скважины не запускает новый расчёт.
12. В режиме `одна кривая → разные скважины` смена кривой не запускает новый расчёт.
13. `Средний портрет кластера` показывает все кластеры, кроме noise, если noise выключен.
14. Цвета кластеров одинаковы во всех режимах и в легенде.
15. Экспорт таблицы интервалов содержит `well_id`, `well_name`, `from_md`, `to_md`, `cluster_id`.
