# План сбора данных для кластерного анализа каротажа

Этот документ описывает, как использовать уже существующую модель БД и текущие UI-экраны для формирования выборки под кластеризацию (электрофации/фации).

## Что уже есть в модели данных

- `well` — скважина: имя, XY, абсолютная отметка (`alt`).
- `well_log` — каротажная кривая скважины: `curve_name`, `curve_data` (JSON массив), `begin`, `end`, `step`, `description`.
- `boundary` — глубинные привязки/маркеры по скважине (`title`, `depth`).
- `well_optionally` — дополнительные атрибуты скважины в ключ-значение формате.

Этого достаточно, чтобы собрать матрицу признаков без изменений схемы БД.

## Какие шаги нужны в UI для подготовки выборки

### 1. Выбор скважин

Источник: таблица `well`.

MVP-подход:
- чекбоксы на списке скважин;
- кнопка "использовать все скважины в текущем объекте" (аналогично логике `all` в окне карты каротажа).

### 2. Выбор набора кривых

Источник: `well_log.curve_name`.

Проблема: разные синонимы названий кривых между скважинами.

MVP-подход:
- экран "Канонизация кривых" (алиасы), где пользователь задает группы имен (например, `GR/GK/GAMMA`);
- при сборке датасета брать первую доступную кривую из группы по каждой скважине.

### 3. Выбор интервала анализа

Источник: `boundary` + ручной диапазон.

Поддержать 3 режима:
- ручной (top/bottom в MD);
- по маркерам (например, между `P2uf GK` и `Xxx-scan`);
- фиксированное окно от выбранной границы (у вас уже есть этот паттерн в расчете медианы по интервалу).

### 4. Ресемплинг по глубине

Источник: `well_log.begin/end/step/curve_data`.

Так как шаги у разных кривых/скважин могут отличаться, нужен единый шаг (например, 0.1/0.2/0.5 м):
- восстановить глубинную ось `begin + i*step`;
- интерполировать выбранные кривые на общий шаг;
- срезать на выбранном интервале.

### 5. Обработка пропусков и фильтрация

MVP-политики:
- drop строк, где отсутствуют N и более признаков;
- медианная импутация по скважине (или по всему набору);
- исключение скважин с покрытием ниже порога (например, < 60%).

### 6. Сбор итоговой таблицы признаков

Итоговая таблица должна содержать:
- `well_id`, `well_name`;
- `depth_md`;
- выбранные признаки (кривые/производные);
- служебно: `boundary_context` (опционально), `source_curve_name` для трассировки.

### 7. Экспорт и воспроизводимость

Сохранять:
- CSV/Parquet матрицу признаков;
- JSON-конфиг сборки (какие скважины, какие алиасы, интервалы, шаг ресемплинга, политика пропусков).

Это позволит переигрывать подготовку данных перед кластеризацией и при переносе на новые скважины.

## Как связать с текущими экранами

- `Well Logging` уже покрывает загрузку LAS/XLS, просмотр, интервал и вычисление агрегатов.
- `MAP` уже покрывает выбор вариантов названий кривых и маркеров + фильтр "all / within object".

Рекомендуется добавить отдельный диалог **"Dataset Builder (Well Clustering)"**, где объединить:
1) выбор скважин;
2) алиасы кривых;
3) режим интервалов;
4) шаг ресемплинга;
5) политику NaN;
6) предпросмотр размера выборки.

## Почему можно начать без миграций БД

Для MVP вся критическая информация уже в БД:
- геометрия и идентичность скважин (`well`),
- сами кривые и их глубинная сетка (`well_log`),
- стратиграфические/рабочие привязки (`boundary`).

Новые таблицы стоит вводить только после стабилизации UX (например, для сохраненных пресетов алиасов и конфигов датасета).
