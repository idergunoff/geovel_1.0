# План патча: отображение результатов кластеризации на радарограмме по профилям

## Цель

Реализовать следующий UX-сценарий без повторного расчета кластеризации:

1. Пользователь нажимает `CALC` в модуле Cluster.
2. После расчета автоматически отрисовывается результат кластеризации на **текущем выбранном профиле** (радарограмма).
3. При смене текущего профиля выполняется **перерисовка из кэша** уже рассчитанных меток, без нового вызова `cluster_data`.

---

## Ключевая идея реализации

Кластеризация остается объектной (по всей выборке `ObjectSet`), но после расчета сохраняется связка:

- `profile_id`
- `trace_index` (индекс измерения внутри профиля)
- `cluster_label`

Эта связка формируется из колонки `prof_index` исходной таблицы (формат `"{profile_id}_{i}"`) и затем используется для отрисовки на радарограмме для любого выбранного профиля.

---

## Ограничения и принципы (чтобы сделать дешевле)

- **Не делаем** новый расчет по каждому профилю.
- **Не делаем** миграции БД в этом патче.
- Держим результат в памяти приложения (runtime-cache).
- Максимально переиспользуем подход отрисовки, похожий на `draw_result_mlp`.

---

### Техническая заметка (этап 0 — выполнено)

- `calculate_cluster()` вызывается только из UI-кнопки `ui.pushButton_clust_calc` (подключение в `geovel.py`), дополнительного автозапуска функции в проекте нет.
- В блоке `geovel.py` уже есть несколько подписок на `ui.comboBox_profile`:
  - `activated.connect(...)` — для обработчиков, завязанных на активный выбор пользователя;
  - `currentTextChanged.connect(...)` — для обновлений списков/связей.
  Для перерисовки кластеров безопаснее использовать `activated` и подключать новый обработчик последним в секции CLUSTER, чтобы не ломать существующую последовательность.
- Источник `ObjectSet.data` для кластера формируется в `collect_clust_object()`:
  1) по профилям собираются таблицы через `build_table.build_table_test(analisis='cluster', curr_form=...)`;
  2) таблицы конкатенируются в `working_data_result`;
  3) в БД сохраняется `ObjectSet(..., data=json.dumps(data))`.
- Формат строк в `build_table_test(..., analisis='cluster')` включает служебные колонки в начале: `['prof_index', 'x_pulc', 'y_pulc']`; далее идут признаки.
- В текущем `clean_features()` служебные поля отбрасываются фиксированно: `X = X[:, 3:]` (первые 3 колонки), значит вход в `cluster_data()` — только численные признаки без метаданных.
- `prof_index` уже готов для будущего маппинга профиля: заполняется как `f"{curr_form.profile_id}_{i}"`.


## Этап 1. Добавить структуру кэша результатов кластеризации

### Что добавить

В `cluster.py` добавить runtime-кэш (модульный уровень):

- `cluster_profile_cache` — словарь вида:

```python
{
  "analysis_key": {
    "profile_labels": {profile_id: {trace_index: label, ...}, ...},
    "meta": {
      "method": ...,
      "n_points": ...,
      "timestamp": ...
    }
  }
}
```

### Что использовать как ключ

`analysis_key` должен однозначно описывать рассчитанный набор. Минимально:

- `clust_object_id`
- `clust_analys_id`
- выбранные параметры расчета (метод, PCA/скейлер, key-параметры алгоритма)

> Упрощенный вариант (для MVP): ключ = `clust_object_id`. Если пользователь жмет `CALC` повторно с новыми параметрами — просто перезаписываем кэш.

### Результат этапа

Есть централизованное место хранения результатов, доступное как из `calculate_cluster`, так и из обработчика смены профиля.

## Этап 2. Сохранить соответствие строк после очистки признаков

### Проблема

Сейчас при `drop_rows` теряется точное соответствие между очищенными строками и исходными `prof_index`.

### Что сделать

Доработать `clean_features(...)`:

1. Вернуть не только очищенные признаки, но и `kept_row_indices`.
2. Для режимов без удаления строк `kept_row_indices = list(range(n_rows))`.
3. Для `drop_rows` вернуть индексы реально сохраненных строк.

### Новый контракт функции

Вариант A (предпочтительный):

```python
X_clean, kept_row_indices = clean_features(...)
```

Вариант B (если боимся затронуть старые вызовы):

```python
result = clean_features(..., return_indices=True)
# result = {"data": X_clean, "kept_row_indices": ...}
```

### Результат этапа

После кластеризации всегда можно восстановить, каким исходным строкам соответствуют метки.

## Этап 3. Построить mapping `profile_id -> trace_index -> label`

### Что сделать в `calculate_cluster()`

1. До очистки зафиксировать исходный массив служебных полей:
   - `raw_meta = np.array(data)[:, 0]` (колонка `prof_index`).
2. После получения `label_list` пройти по индексам очищенного набора и через `kept_row_indices` достать исходный `prof_index`.
3. Распарсить `prof_index` в `(profile_id, trace_index)`.
4. Собрать словарь `profile_labels`:

```python
profile_labels[profile_id][trace_index] = label
```

5. Сохранить `profile_labels` в runtime-кэш.

### Валидации

- Если `prof_index` не парсится, логировать предупреждение и пропускать запись.
- Если дубликат `(profile_id, trace_index)`, оставлять последний label и писать `set_info` warning.

### Результат этапа

После каждого `CALC` есть кэш меток, пригодный для отрисовки на любом профиле.

## Этап 4. Реализовать отрисовку кластеров на радарограмме для текущего профиля

### Новая функция

Добавить функцию вида:

```python
def draw_cluster_profile_result(profile_id: int, profile_labels: dict, *, use_relief=True):
    ...
```

### Логика

1. Найти текущую формацию/границы (`list_up`, `list_down`) аналогично логике в `draw_result_mlp`.
2. Собрать последовательность меток по трассам текущего профиля:
   - если для индекса нет метки (например, удалена строка), ставить `None` и пропускать заливку.
3. Разбить последовательность на непрерывные сегменты одинакового label.
4. Для каждого сегмента вызвать `draw_fill_result` своим цветом.
5. Для `label == -1` использовать нейтральный серый цвет (шум).

### Цвета

- Сделать helper `get_cluster_color(label)`:
  - для `-1`: `#808080`
  - для остальных: циклическая палитра из 10–20 цветов.

### Результат этапа

Сразу после `CALC` пользователь видит “полосы кластеров” на радарограмме выбранного профиля.

## Этап 5. Перерисовка при смене профиля без пересчета

### Что добавить

1. Новый обработчик:

```python
def redraw_cluster_for_current_profile_from_cache():
    ...
```

2. Подписка в `geovel.py` на событие смены профиля:

- `ui.comboBox_profile.activated.connect(redraw_cluster_for_current_profile_from_cache)`
  или более безопасный сигнал, уже используемый проектом.

### Логика обработчика

1. Проверить наличие актуального кэша.
2. Взять `profile_id` текущего профиля.
3. Если для профиля есть метки — вызвать `draw_cluster_profile_result(...)`.
4. Если нет меток — мягко показать `set_info("Для текущего профиля нет кластерных меток в кэше", "brown")`.

### Анти-рекурсия

Если смена профиля внутри других обработчиков может дергать перерисовку много раз:

- добавить простой guard-флаг `is_cluster_redraw_in_progress`.

### Результат этапа

## Этап 6. UX и устойчивость

### UX-доработки (минимум)

1. После `CALC` писать в `set_info`, что доступна перерисовка по профилям без пересчета.

### Результат этапа

## Этап 7. Тест-план (ручной, обязательный)

### Сценарии

1. **Базовый happy-path**
   - Собрать `ObjectSet`.
   - Нажать `CALC`.
   - Убедиться, что карта рисуется как раньше и одновременно появилась заливка на текущем профиле.

2. **Смена профиля**
   - После `CALC` переключить профиль 3–5 раз.
   - Проверить, что перерисовка идет мгновенно и без пересчета.

3. **Режим очистки `drop_rows`**
   - Включить удаление строк с NaN.
   - Проверить, что отрисовка не падает и метки корректно ложатся на профиль.

4. **Профиль без меток**
   - Переключиться на профиль, который не вошел в объектный набор.
   - Проверить корректное предупреждение в `set_info`.

5. **Повторный `CALC` с другими параметрами**
   - Пересчитать кластеризацию другим методом.
   - Проверить, что кэш обновился и перерисовка использует новые метки.

6. **Шум HDBSCAN**
   - Убедиться, что `-1` отображается отдельным цветом (серым).

## Риски и как снизить

1. **Нарушение совместимости `clean_features`**
   - Снизить риск через опциональный параметр `return_indices`.

2. **Конфликты с текущими обработчиками `comboBox_profile`**
   - Подключать перерисовку последним обработчиком и добавить guard-флаг.

3. **Некорректный парсинг `prof_index`**
   - Защитный `try/except` + предупреждение в `set_info`.

4. **Визуальные артефакты на рельефе/границах**
   - Переиспользовать текущую проверенную логику границ из MLP-отрисовки.

---

## Критерии приемки патча

1. После `CALC` всегда отображается кластерная заливка на текущем профиле.
2. Смена профиля перерисовывает заливку из кэша без повторного `cluster_data`.
3. При `drop_rows` визуализация не ломается.
4. Старое поведение карты кластеров сохранено.
5. Ошибки/пустые кейсы отрабатываются сообщениями, без падения UI.
