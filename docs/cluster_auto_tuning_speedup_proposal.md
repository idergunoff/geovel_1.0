# Предложение: безопасное ускорение AUTO-подбора в кластерном анализе

## Контекст

В `cluster.py` уже есть важные защитные механизмы (ограничение выборки, кэш преобразований, лимиты по памяти и таймауты). Ниже — следующий набор шагов, который ускоряет подбор без потери воспроизводимости и с минимальным риском регрессий.

## 1) Быстрый выигрыш (минимальный риск)

1. **Ввести early-stop по «плато качества» в COARSE**  
   Если за последние `N` валидных кандидатов лучший `score` не улучшился более чем на `eps`, завершать COARSE досрочно.
   - Безопасность: включать только при `checkBox_cluster_auto_total_time=True` или отдельным флагом `auto_early_stop`.
   - Профит: сокращение 15–40% кандидатов на «плоских» наборах.

2. **Сделать адаптивный размер silhouette-подвыборки**  
   Сейчас есть фиксированные лимиты; лучше добавить формулу:  
   `sil_samples = min(max_limit, max(min_limit, int(alpha * sqrt(n_rows) * n_clusters)))`.
   - Безопасность: метрики CH/DB считаются на полной подвыборке, silhouette остаётся детерминированной при том же `seed`.
   - Профит: заметное ускорение на больших выборках.

3. **Ужать FINE для слабых seed-кандидатов**  
   После COARSE брать в FINE только кандидаты в пределах `delta_score` от лидера (например, `<= 0.03`).
   - Безопасность: лидер и ближайшие конкуренты всегда остаются.
   - Профит: минус 20–60% fine-кандидатов.

## 2) Средний риск, высокий эффект

4. **Bandit/Successive Halving для кандидатов**  
   Двухфазная схема:
   - Фаза A: быстрый прогон кандидатов на небольшой подвыборке.
   - Фаза B: топ `p%` прогоняется на полной подвыборке.
   - Безопасность: итоговый лидер выбирается только по фазе B.

5. **Тонкая стратифицированная подвыборка вместо чисто случайной**  
   Для 1D/2D геопрофилей применить бакетизацию по глубине/координате и брать равные доли из бакетов.
   - Безопасность: меньше риск «потерять» редкие геологические зоны.
   - Профит: стабильнее и быстрее convergence у AUTO.

6. **Кэш на уровне «алгоритм+параметры+X_hash» для метрик**  
   Если один и тот же `labels` уже встречался (или близкий hash), переиспользовать CH/DB/silhouette.
   - Безопасность: использовать только при точном совпадении hash.
   - Профит: особенно полезно в FINE и rescue.

## 3) Что не делать (или делать только опционально)

- Не отключать полностью CH/DB: ухудшается устойчивость ранжирования.
- Не включать агрессивный параллелизм `n_jobs=-1` везде: можно получить OOM/фризы UI.
- Не менять одновременно search-space и scoring: сложно объяснить, что именно дало эффект.

## 4) Порядок внедрения (рекомендуемый)

1. Early-stop COARSE + delta-filter в FINE.
2. Адаптивный silhouette sampling.
3. Successive Halving (feature-flag `auto_halving`).
4. После этого — только при необходимости: metric-cache по hash labels.

## 5) Набор проверок после каждого шага

1. **Корректность**: нет падений, есть минимум 1 `status=ok` на тестовых ObjectSet.
2. **Стабильность**: при одинаковом `seed` одинаковый top-1/top-3 (допуск по перестановке эквивалентных score).
3. **Качество**: деградация best-score не хуже `-3%` относительно baseline.
4. **Скорость**: медиана времени AUTO (10 прогонов) лучше baseline минимум на `20%`.

## 6) Минимальные целевые KPI

- COARSE: `-25%` времени без ухудшения top-1 качества.
- FINE: `-35%` числа прогонов кандидатов.
- End-to-end AUTO FINE: `-30%` wall-clock на больших наборах (`n_rows > 10k`).

## 7) Практический старт (без переписывания архитектуры)

- Добавить 3 параметра в UI/конфиг:
  - `auto_early_stop_patience` (по умолчанию 20)
  - `auto_early_stop_min_delta` (по умолчанию 0.005)
  - `auto_fine_delta_from_best` (по умолчанию 0.03)
- Логировать в `set_info` причины досрочного остановa и объём срезанных кандидатов.

Это даст быстрый, безопасный и измеримый прирост производительности, сохраняя текущий pipeline и UX.
